Core Managers

The gameplay managers that implement targeting, pathfinding, movement prediction, macro execution, the journal, parties, and corpses — all owned by the World instance and updated each frame by GameScene.

The Manager Pattern

TazUO uses constructor-injected managers rather than singletons (with a few exceptions). Every manager is instantiated in World()'s constructor, receives the World reference, and is exposed as a property: 

// World constructor creates all managers
public World()
{
    CorpseManager  = new CorpseManager(this);
    Party          = new PartyManager(this);
    TargetManager  = new TargetManager(this);
    Macros         = new MacroManager(this);
    Journal        = new JournalManager();
    // ...15+ more managers
}

Most managers expose an Update(uint ticks) or Update(double totalTime, double frameTime) method that is called from GameScene.Update(). They react to incoming network data (set by packet handlers) and drive outgoing actions (sending packets or mutating world state).

TargetManager

Game/Managers/TargetManager.cs — manages the cursor targeting system. When the server requests a target (0x6C), or a macro/script initiates targeting, TargetManager tracks what kind of target is expected and routes the user's click to the right network packet.

Enum / PropertyMeaning
CursorTarget.ObjectStandard single-click target on an entity
CursorTarget.PositionTarget a tile position (used for spells like Teleport)
CursorTarget.MultiPlacementPlacing a house or ship deeded structure
CursorTarget.CallbackTargetClient-side target that fires a C# delegate instead of a packet
TargetType.Neutral / Harmful / BeneficialCursor colour hint (blue/red/green crosshair)
 
// Key public API
public bool   IsTargeting   { get; }      // cursor is in target mode
public CursorTarget TargetingType { get; }
public uint   LastTargetInfo { get; set; }  // serial of last selected target
public uint   NewTargetSystem { get; set; } // TazUO: persisted "attack this" serial

public void SetTargeting(CursorTarget type, TargetType targetType, uint cursorID) { ... }
public void Target(uint serial) { ... }    // select object target, sends 0x6C response
public void Target(ushort x, ushort y, sbyte z, ushort graphic) { ... } // position target
public void CancelTarget() { ... }         // sends cancel 0x6C to server
Last Target vs New Target System
Classic UO has "last target" (repeatable via macro). TazUO adds a new target system (NewTargetSystem) — a separate persistent target serial that HealthLinesManager uses to draw a special indicator on the selected target. It survives longer than the temporary last-target and is used by combat-assist features.

Pathfinder — A* Auto-Walk

Game/Pathfinder.cs — implements A* pathfinding over the UO world tile grid. It is owned by PlayerMobile (accessible via World.Player.Pathfinder), not by World directly. 

public sealed class Pathfinder
{
    private const int PATHFINDER_MAX_NODES = 150000; // A* node budget
    private static PriorityQueue _openSet;
    private static Dictionary<(int,int,int), PathNode> _closedSet;
    private static List<PathNode> _path;      // the computed path

    public bool  AutoWalking        { get; set; }   // true while walking along path
    public bool  BlockMoving        { get; set; }
    public Point StartPoint         { get; }
    public Point EndPoint           { get; }
    public int   PathSize           => _path.Count;

    public static bool PathFindingCanBeCancelled { get; set; }
    public static bool FastRotation             { get; set; }
    public bool        UseLongDistancePathfinding { get; set; }
}

The 8-directional movement offsets (_offsetX / _offsetY) support diagonal movement. The node budget of 150,000 prevents runaway pathfinding on impossible routes. Pathfinding results are consumed by WalkerManager which translates path steps into actual walk requests sent to the server.

WalkerManager — Movement Prediction

Game/Managers/WalkerManager.cs — handles client-side movement prediction for the local player. UO clients send walk requests (0x02/0x97) to the server with a sequence number, then immediately move the player locally. The server sends 0x22 ConfirmWalk or 0x21 DenyWalk to accept/reject each step.

Struct/TypePurpose
StepInfoOne pending step: Direction, Sequence number, Accepted flag, X/Y/Z destination
FastWalkStackAnti-cheat validation keys from server; must be included in walk packets

The prediction stack holds at most Constants.MAX_STEP_COUNT pending steps. If the server denies a step, the walker snaps the player back to the server-confirmed position and flushes all pending steps. This is the "rubber-band" effect seen when walking into walls.

MacroManager

Game/Managers/MacroManager.cs — records and plays back macros. A macro is a linked list of MacroObject nodes, each describing one action (cast spell, use skill, use item, say text, etc.). 

public sealed class MacroManager : LinkedObject
{
    // Macro execution state
    private MacroObject _lastMacro;   // currently executing node
    private long _nextTimer;           // Time.Ticks when to advance

    // Skill index table — maps macro skill slot → UO skill index
    private readonly byte[] _skillTable = { 1, 2, 35, 4, ... };

    // Spellbook sizes for each magic school
    private readonly int[] _spellsCountTable = { 64, 17, ... };

    public long WaitForTargetTimer { get; set; }  // used by "WaitForTarget" macro action
    public bool WaitingBandageTarget { get; set; }
}

Macro Storage

Macros are stored as XML files in Data/Macros/. Each Macro has:

  • A name and key binding (SDL keycode + modifier flags)
  • A linked list of MacroObject nodes
  • Each node has a MacroType enum value and optional subcode / text

Macro Execution

MacroManager.Update() is called each frame. If a macro is running, it evaluates the current MacroObject against Time.Ticks. Actions that require server confirmation (e.g. "Use Skill") set WaitForTargetTimer to pause execution until the server responds. The integration with LegionScripting means Python scripts can also trigger macros by name.

JournalManager

Game/Managers/JournalManager.cs — the chat and message log. All in-game text messages (NPC speech, player chat, system messages, combat text) flow through Journal.Add(). 

// Journal is a bounded deque — oldest entries are discarded
public static Deque<JournalEntry> Entries { get; }
    = new(Constants.MAX_JOURNAL_HISTORY_COUNT);

public void Add(string text, ushort hue, string name,
               TextType type, bool isunicode, MessageType messageType)
{
    if (JournalFilterManager.Instance.IgnoreMessage(text)) return;

    // Recycle oldest entry if at capacity
    JournalEntry entry = Entries.Count >= MAX_JOURNAL_HISTORY_COUNT
        ? Entries.RemoveFromFront()
        : new JournalEntry();

    // Fill entry fields (text, hue, name, font, time, type)
    Entries.AddToBack(entry);
    EventSink.InvokeJournalEntryAdded(null, entry);

    // Optionally write to disk (_fileWriter, lazy-initialized)
}

Scripts (LegionScripting) subscribe to journal events via EventSink.JournalEntryAdded to implement pattern-matching bots. The JournalFilterManager allows the user to suppress specific messages by regex pattern.

PartyManager

Game/Managers/PartyManager.cs — tracks party membership and provides data for party HP bars. 

public sealed class PartyManager
{
    public bool  InParty  => Leader != 0;
    public uint  Leader   { get; set; }   // serial of party leader
    public uint  Inviter  { get; set; }   // serial of pending invite sender
    public bool  CanLoot  { get; set; }   // can loot party members' corpses

    public PartyMember[] Members { get; } = new PartyMember[10]; // max 10

    // Auto-heal support for macro "heal party member"
    public long PartyHealTimer  { get; set; }
    public uint PartyHealTarget { get; set; }
}

ParsePacket() handles the 0xBF extended party packets. Party updates create or destroy BaseHealthBarGump instances for each member, giving the classic party HP bar display. InParty is used by combat macros to determine valid heal targets.

CorpseManager

Game/Managers/CorpseManager.cs — maps corpse Item serials back to the Mobile that died. This is needed for equipment display (corpses can be searched) and for auto-loot (need to know whose corpse it is). 

public sealed class CorpseManager
{
    private readonly Deque<CorpseInfo> _corpses;

    public void Add(uint corpse, uint obj, Direction dir, bool run) { ... }
    public void Remove(uint corpse, uint obj) { ... }
    public CorpseInfo GetCorpseObject(uint serial) { ... }
}

public struct CorpseInfo
{
    public uint      CorpseSerial;  // the Item serial of the corpse
    public uint      ObjectSerial;  // the Mobile that died
    public Direction Direction;     // direction at death (for equipment display)
    public bool      IsRunning;     // was the mobile running when it died?
}

When a mobile dies (0x2C DeathScreen), CorpseManager.Add() is called pairing the new corpse item with the mobile's serial. When the corpse decays (0x1D Delete), CorpseManager.Remove() cleans up the entry. GetCorpseObject() is the lookup path used by AutoLootManager to find whose corpse to open.

Going Forward

Adding a New Manager

  1. Create Game/Managers/MyManager.cs. Accept World world in the constructor and store it.
  2. Add a property public MyManager MyManager { get; } to World.cs.
  3. Instantiate in World()'s constructor: MyManager = new MyManager(this);
  4. Call world.MyManager.Update(ticks) from GameScene.Update() if the manager needs per-frame processing.
  5. Subscribe to EventSink events if the manager needs to react to specific game events (player created, entity destroyed, etc.).

EventSink — The Event Bus

EventSink is a static class with event Action<T> fields for cross-cutting concerns:

  • EventSink.OnPlayerCreated — fired when World.Player is set
  • EventSink.JournalEntryAdded — fired on every journal message (used by scripting)
  • EventSink.OnPlayerStatChange — fired when player HP/Mana/Stamina changes
  • EventSink.SkillValueChangedEvent — on skill gain/loss
Manager lifetime
Managers are tied to the World instance. When the player logs out or the scene changes, a new World is created and all managers are re-instantiated. Static state in managers will persist across scene changes — be careful with static fields that hold world-object references.
