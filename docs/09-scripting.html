09 — Scripting System (LegionScripting)

TazUO embeds a full IronPython 3 runtime so players can automate gameplay with Python scripts. The system — called LegionScripting — exposes almost the entire game API as thread-safe Python-callable C# objects. This document covers the runtime architecture, script lifecycle, every major class in the hierarchy, the event system, gump creation, persistent variables, and how to extend the API.

IronPython Runtime

Runtime Version & Libraries

TazUO bundles IronPython 3.4.2 in external/iplib/. The runtime is pre-loaded in the background at game startup to reduce the latency of the first script execution:

// LegionScripting.cs — Init()
Task.Factory.StartNew(Python.CreateEngine); // warm-up

Each script gets its own isolated engine instance (via Python.CreateEngine()) unless the module cache is enabled, in which case the engine is reused between runs of the same script. The engine's search paths include:

  • iplib/ — IronPython standard library
  • LegionScripts/ — user script root (for local imports)
  • The directory of the script being executed

Threading Model

Critical: Scripts run on their own thread, not the game main thread. Every access to game state from Python must be marshalled via MainThreadQueue.InvokeOnMainThread() or MainThreadQueue.EnqueueAction(). The Py* wrapper classes do this automatically for every property and method they expose.
Python Script Thread Game Main Thread ──────────────────── ──────────────── API.Player.Hits MainThreadQueue processes → MainThreadQueue.InvokeOnMainThread(...) → blocks until result ready → executes lambda ← returns int → writes result to TaskCompletionSource GameActions.Walk(...) → MainThreadQueue.EnqueueAction(...) → fire-and-forget → executes lambda on next Update()

InvokeOnMainThread() blocks the script thread using a TaskCompletionSource — it is safe to call from Python but avoid calling it inside a tight loop without a pause, as it will starve the game thread. Use API.Pause(seconds) to yield.

Script Lifecycle

Loading vs Playing

Scripts are loaded (registered in LoadedScripts) when discovered on disk — this just reads the file and creates a ScriptFile instance. They are played (executed) separately:

LegionScripting.Init(world) ├─ Task.Factory.StartNew(Python.CreateEngine) warm-up engine ├─ LoadScriptsFromFile() walk LegionScripts/ │ ├─ Up to 2 directory levels deep │ └─ .py / .lscript → new ScriptFile(world, path, name) ├─ LoadLScriptSettings() lscript.json ├─ AutoPlayGlobal() → PlayScript(f) for each └─ AutoPlayChar() → PlayScript(f) for each PlayScript(script) ├─ script.ReadFromFile() re-reads source │ └─ strips "import API" lines (API is injected into builtins) ├─ new Thread(() => ExecutePythonScript(script)) ├─ PyThreads[threadId] = script for lookup by thread └─ thread.Start() ExecutePythonScript(script) [script's own thread] ├─ script.SetupPythonEngine() Python.CreateEngine() ├─ script.SetupPythonScope() │ ├─ PythonScope = engine.CreateScope() │ ├─ new API(engine) → ScopedApi │ └─ builtins["API"] = api makes API a builtin ├─ source = engine.CreateScriptSourceFromString(contents, path) ├─ source.Execute(scope) └─ [on exit] MainThreadQueue.EnqueueAction(StopScript) StopScript(script) ├─ api.StopRequested = true ├─ api.CancellationToken.Cancel() ├─ engine.Runtime.Shutdown() ├─ thread.Interrupt() throws ThreadInterruptedException └─ [on thread exit] PythonScriptStopped() ├─ api.CloseGumps() ├─ api.Dispose() └─ scope = null

Error Handling

When a script throws an unhandled exception, ShowScriptError() formats it using IronPython's ExceptionOperations.FormatException(). It then parses the formatted traceback with a regex to extract file paths and line numbers, reads the source file, and opens a ScriptErrorWindow (ImGui) with context lines around the error. The line number is offset by 1 because the import API line was stripped from the source.

The API Object

API is the single access point for every script. It is injected as a Python builtin so scripts can use API.SomeMethod() without any import. Each running script gets its own isolated API instance (ScopedApi), so one script's journal queue never mixes with another's.

Callback / Hotkey Architecture

Hotkey callbacks cannot call game methods directly because they fire from the keyboard event handler (which may run on a different thread). Instead they are put into a Queue<Action> scheduledCallbacks and drained in order by the script's own loop when it calls API.ProcessCallbacks():

# Typical event loop pattern in a script
def on_shift_a():
    API.SysMsg("SHIFT+A pressed!")

API.OnHotKey("SHIFT+A", on_shift_a)

while not API.StopRequested:
    API.ProcessCallbacks()   # drains the callback queue
    API.Pause(0.1)

Shared variables (SetSharedVar/GetSharedVar) use a static ConcurrentDictionary<string, object> — they survive across multiple script instances. Persistent variables are backed by SQLite (see below).

GetEntity() (protected) checks the cached reference — if the serial still matches, returns it directly. Otherwise calls World.Get(Serial) via MainThreadQueue. This means short-lived Python objects don't cause dangling pointer problems even if the game object is recycled.

Event System

Events are accessed through API.Events (a PyEvents instance). Each method registers a Python callback to fire when the corresponding game event occurs. Callbacks are delivered via ScheduleCallback() and must be drained by API.ProcessCallbacks() in the script's main loop.

Available Events

MethodCallback ReceivesDescription
API.Events.OnPlayerHitsChanged(cb)int new_hitsPlayer HP changed
API.Events.OnBuffAdded(cb)Buff objectBuff/debuff applied
API.Events.OnBuffRemoved(cb)Buff objectBuff/debuff removed
API.Events.OnPlayerDeath(cb)uint serialPlayer died
API.Events.OnOpenContainer(cb)uint serialContainer gump opened
API.Events.OnPlayerMoved(cb)PositionChangedArgsPlayer moved one tile; .NewLocation available
API.Events.OnItemCreated(cb)uint serialNew item added to world
Journal entries are delivered differently — they are pushed into a per-script ConcurrentQueue<PyJournalEntry> by the global EventSink.JournalEntryAdded handler in LegionScripting.cs. Scripts drain it with: 
while API.JournalEntries.Count > 0:
    entry = API.JournalEntries.Dequeue()
    if "You are poisoned" in entry.Text:
        API.UseItem(curePotion)
The queue is capped at Profile.MaxJournalEntries; oldest entries are silently dropped when full.

Event Implementation Pattern ([GenApiEvent])

PyEvents is a partial class. Each event is declared with the [GenApiEvent("EventSinkEventName")] attribute on a partial method. A source generator (or build-time equivalent) produces the subscription/unsubscription boilerplate, wiring the C# EventSink event to call _api.ScheduleCallback(callback, args...).

// PyEvents.cs — declaration only
[GenApiEvent("OnPositionChanged")]
public partial void OnPlayerMoved(object callback);

// Generated glue (conceptual)
public partial void OnPlayerMoved(object callback)
{
    EventSink.OnPositionChanged += (sender, args) =>
        _api.ScheduleCallback(callback, args);
}

Custom Gumps from Python

Scripts can create fully interactive UI windows using API.Gumps (PyGumps). All gumps created through the API are tracked in api.gumps (a ConcurrentBag<Gump>) and are automatically closed when the script stops (unless keepOpen=True).

Quick Example

# Create a movable window with a label and button
g = API.Gumps.CreateGump(acceptMouseInput=True, canMove=True)
g.SetRect(100, 100, 200, 120)

label = API.Gumps.CreateGumpLabel("Hello World!", hue=0x34)
label.SetRect(10, 10, 180, 30)
g.Add(label)

btn = API.Gumps.CreateGumpButton("Click Me")
btn.SetRect(60, 60, 80, 30)
g.Add(btn)

API.Gumps.AddGump(g)

while not API.StopRequested:
    API.ProcessCallbacks()
    if btn.HasBeenClicked():
        API.SysMsg("Button clicked!")
    API.Pause(0.1)

Persistent Variables

LegionScripting provides cross-session, SQLite-backed variable storage via PersistentVars. Variables are namespaced by scope:

ScopeKey = (enum)Scope Key
GlobalAPI.PersistentVar.Global"GLOBAL" — same across all characters and servers
ServerAPI.PersistentVar.ServerServerName
AccountAPI.PersistentVar.AccountServerName + Username
CharacterAPI.PersistentVar.CharServerName + Username + CharName

Storage is in Data/legionvars.db. Access from Python:

# Save and retrieve a character-scoped variable
API.SetPersistentVar(API.PersistentVar.Char, "lastX", str(API.Player.X))

x = API.GetPersistentVar(API.PersistentVar.Char, "lastX", "0")
API.SysMsg(f"Last X: {x}")
Storage is asynchronous: SaveVar calls SaveVarAsync(...).ConfigureAwait(false) so it doesn't block the script thread. The DB is protected by a SemaphoreSlim(1,1). An old legionvars.dat flat-file format is automatically migrated to SQLite on first run and backed up as .bak.

Auto-Generated Documentation

The src/APIToMarkdown/ project uses Roslyn (Microsoft.CodeAnalysis.CSharp) to parse each API.cs and PyClasses/*.cs file at build time and emit Markdown + a stub API.py file:

Outputs live in src/ClassicUO.Client/LegionScripting/docs/:

  • API.md — main API class (properties, methods, enums)
  • PyPlayer.md, PyMobile.md, PyItem.md, etc. — one file per class
  • Events.md, Gumps.md, PyBaseControl.md — sub-object docs
  • notes.md — handwritten additional notes

Using the API.py stub: Place API.py in the same folder as your scripts, add import API (or from API import *) at the top, and VS Code / PyCharm will offer autocompletion for all API methods. The import line is automatically stripped at execution time — it has no runtime cost.

Script Browser

ScriptBrowser is a SingletonImGuiWindow that lets players browse the public PlayTazUO/PublicLegionScripts GitHub repository and download scripts directly in-game. It uses the GitHub Contents API (api.github.com/repos/…/contents/…) with a TTL-based GitHubContentCache. Downloaded scripts are saved to LegionScripts/ and immediately available.

Adding a New Python API Method

To expose a new game capability to scripts:

Step 1 — Add the C# method to API.cs

/// <summary>
/// Shout a message in the game world.
/// Example:
/// ```py
/// API.Shout("For honor!")
/// ```
/// </summary>
/// <param name="text">The text to shout</param>
public void Shout(string text)
    => MainThreadQueue.InvokeOnMainThread(
        () => GameActions.Say(World, text, MessageType.Yell, Constants.SPEECH_HUE)
    );
Always use MainThreadQueue. Game state is single-threaded. Calling GameActions.* directly from the script thread will cause race conditions or crashes.
  • Use InvokeOnMainThread() when you need a return value (blocks the script thread until complete)
  • Use EnqueueAction() for fire-and-forget operations

Step 2 — (Optional) Add a new event to PyEvents.cs

/// <summary>Called when player says something.</summary>
[GenApiEvent("OnSpeech")]
public partial void OnSpeech(object callback);

The [GenApiEvent] attribute tells the source generator to wire the EventSink.OnSpeech event → ScheduleCallback(callback, args) automatically.

Step 3 — Rebuild and regenerate docs

dotnet build -c Release

The APIToMarkdown build step will pick up the new XML doc comment and add it to the generated Markdown and API.py stub.

Step 4 — Test from Python

# Test in a minimal script
API.Shout("For honor!")
API.Pause(1)

Going Forward — Safe Patterns

Standard script loop template: 
while not API.StopRequested:
    # 1. Process UI callbacks (button clicks, event callbacks)
    API.ProcessCallbacks()

    # 2. Read game state (auto-marshalled via MainThreadQueue)
    player = API.Player
    if player.HitsDiff > 20:
        API.BandageSelf()

    # 3. Always pause — never spin-lock
    API.Pause(0.25)
Common pitfalls to avoid:
  • No bare spin-loops — always call API.Pause(). A tight loop starves the main thread because InvokeOnMainThread contends for the same dispatcher queue.
  • Don't store PyEntity references long-term — game objects can be destroyed. Re-query each loop iteration or check entity.IsDestroyed via GetEntity().
  • Check API.StopRequested in all loops. This is the clean shutdown signal. Scripts that ignore it will be killed by Thread.Interrupt() which throws ThreadInterruptedException at the next blocking call.
  • Don't call game APIs from event callbacks directly — they fire from the main thread via ScheduleCallback; use ProcessCallbacks() to pick them up on the script thread, then call game methods.
  • Shared vars (SetSharedVar) use a static dictionary — they persist until explicitly cleared with ClearSharedVars(). Large objects held in shared vars will prevent GC.
  • Don't call blocking IO inside InvokeOnMainThread — it blocks the entire game frame. Do async IO outside the main-thread call.
Performance tip: Reading many properties in sequence each incurs a round-trip through MainThreadQueue. Batch reads by building a snapshot object on the main thread: 
def get_snapshot():
    p = API.Player
    return {'hits': p.Hits, 'mana': p.Mana, 'stam': p.Stamina}

snap = get_snapshot()   # 3 invocations at once is fine
