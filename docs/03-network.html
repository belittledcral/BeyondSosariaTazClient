Network & Packet System

How TazUO connects to the UO server, the two-thread architecture for send and receive, packet parsing and opcode dispatch, outgoing packet construction, encryption modes, and the critical thread-safety rules every developer must follow.

Architecture Overview

The network layer is split across two files and one subdirectory:

FilePurpose
Network/AsyncNetClient.csAsyncSocketWrapper (TCP wrapper) + AsyncNetClient (protocol layer). The singleton is AsyncNetClient.Socket.
Network/PacketHandlers/HandlerRegistry.csFull opcode-to-handler registration list.
Network/PacketHandlers/PacketParser.csParses raw bytes from the circular buffer and dispatches to handlers.
Network/PacketHandlers/*.cs~100 individual handler files, one per packet type or logical group.
Network/OutgoingPackets.csExtension methods on AsyncNetClient that build and send every outgoing packet type.
Network/Encryption/Encryption.csEncryption helper; derives keys from client version, delegates to Blowfish/Twofish/LoginCrypt.
Network/PacketsTable.cs256-entry table mapping opcode → fixed packet length (−1 = variable length).

Connection Lifecycle

Login flow ─────────────────────────────────────────────────────────── 1. LoginScene.ConnectToLoginServer() └─ await AsyncNetClient.Socket.Connect(ip, port) │ ├─ AsyncSocketWrapper.ConnectAsync() — TCP connect (2 s timeout) ├─ Task.Run(ReceiveLoopAsync) — background receive task starts └─ Task.Run(NetworkLoopAsync) — background send task starts 2. Client sends 0xEF SeedRequest (new handshake) or 0x00 Login Seed (depends on client version / server flags) 3. Server sends 0xA8 Server List → LoginScene shows server list 4. Client sends 0xA0 Select Server 5. Server sends 0x8C Server Relay (new IP:port for game connection) 6. LoginScene reconnects to game server → sends 0x91 GameLogin 7. Server sends 0x5D Play Character → client sends 0x5D back 8. Server sends 0x1B Enter World → World is created, map loaded, GameScene started 9. Server sends 0x55 Login Complete → all done; normal play begins ───────────────────────────────────────────────────────────

The single AsyncNetClient.Socket instance is reused across the login → relay reconnect sequence. Connect() clears the send buffer and resets Huffman state before each connection attempt.

Threading Model

TazUO uses three threads for networking. Understanding which data each thread touches is essential to writing safe code.

┌──────────────────────────────────────────────────────────┐ │ RECEIVE TASK (Task.Run, thread pool) │ │ │ │ AsyncSocketWrapper.ReceiveLoopAsync() │ │ → await stream.ReadAsync(buffer, ...) — blocks here │ │ → client.OnDataReceived(data) │ │ → ProcessEncryption(span) decrypt in-place │ │ → DecompressBuffer(span) Huffman decompress │ │ → _incomingMessages.Enqueue(bytes) ← ConcurrentQueue │ └──────────────────────────────────────────────────────────┘ ┌──────────────────────────────────────────────────────────┐ │ SEND TASK (Task.Run, thread pool) │ │ │ │ AsyncNetClient.NetworkLoopAsync() │ │ → await _sendSignal.WaitAsync(50ms) — blocks here │ │ → ProcessSendAsync() │ │ → lock(_sendStream) { drain CircularBuffer } │ │ → await socket.SendAsync(chunk) │ └──────────────────────────────────────────────────────────┘ ┌──────────────────────────────────────────────────────────┐ │ MAIN THREAD (FNA game loop) │ │ │ │ GameController.Update() │ │ → while (Socket.TryDequeuePacket(out byte[] pkt)) │ │ PacketParser.Instance.ParsePackets(world, pkt) │ │ → _handlers[opcode](world, ref reader) │ │ → mutates World.Mobiles / Items etc. │ └──────────────────────────────────────────────────────────┘
Critical thread-safety rule
Only the main thread may read or write World.Mobiles, World.Items, World.Player, or any GameObject field. The receive task only enqueues raw byte arrays into a ConcurrentQueue. The main thread dequeues and dispatches during Update(). Never access world state from the receive or send tasks.

Receive Pipeline in Detail

// AsyncSocketWrapper.ReceiveLoopAsync (background thread)
byte[] buffer = ArrayPool<byte>.Shared.Rent(4096);
while (!cancellationToken.IsCancellationRequested && IsConnected)
{
    int bytesRead = await _stream.ReadAsync(buffer, 0, buffer.Length, ct);
    if (bytesRead == 0) { Disconnect(); break; }
    // Copy to heap (the handler runs on main thread later)
    _client.OnDataReceived(data);
}

// AsyncNetClient.OnDataReceived — still background thread
void OnDataReceived(byte[] data)
{
    ProcessEncryption(data.AsSpan());    // decrypt in-place
    Span<byte> decompressed = DecompressBuffer(data.AsSpan());
    _incomingMessages.Enqueue(decompressed.ToArray());  // ConcurrentQueue
}

// Main thread — GameController.Update()
while (AsyncNetClient.Socket.TryDequeuePacket(out byte[] pkt))
    PacketParser.Instance.ParsePackets(world, pkt.AsSpan());

The ReceiveLoopAsync calls ReadAsync which blocks until data arrives — no polling. The Huffman decompressor (DecompressBuffer) is enabled after the server sends 0x80 Compression Enable or after the login handshake for newer clients.

Send Pipeline in Detail

// AsyncNetClient.Send() — called from main thread
public void Send(Span<byte> message, ...)
{
    // 1. Optional plugin intercept
    if (!ignorePlugin && Plugin.Enabled && !Plugin.ProcessSendPacket(ref message)) return;

    // 2. Encrypt (in-place, Span overwrite)
    EncryptionHelper.Instance?.Encrypt(!_isCompressionEnabled, message, message, message.Length);

    // 3. Enqueue to circular buffer (locked)
    lock (_sendStream) { _sendStream.Enqueue(message); }

    // 4. Signal the send task
    _sendSignal.Release();
}

// NetworkLoopAsync — background send task
while (!ct.IsCancellationRequested && IsConnected)
{
    await _sendSignal.WaitAsync(50, ct);    // blocks until Send() signals
    await ProcessSendAsync(ct);              // drain CircularBuffer → stream.WriteAsync
    Statistics.Update();
}

The 50 ms timeout in WaitAsync ensures statistics are updated even when no packets are being sent. The send task drains the entire CircularBuffer each wake-up, so multiple packets enqueued between ticks are coalesced into fewer WriteAsync calls.

Packet Structure

Every UO packet begins with a 1-byte opcode. Whether the packet has a fixed or variable length depends on the opcode:

Fixed-length packet (e.g. 0x73 Ping — 2 bytes): [ 0x73 ] [ data... ] ↑ opcode (total length known from PacketsTable) Variable-length packet (e.g. 0x1C Talk — many bytes): [ opcode ] [ length hi ] [ length lo ] [ data... ] ↑ ↑ big-endian ushort total length including these 3 bytes

PacketsTable holds a short[256] indexed by opcode. A value ≥ 0 is the fixed packet length. A value of −1 means the next two bytes (bytes 1–2) are a big-endian ushort total length. The parser uses this to know how many bytes to accumulate before dispatching. 

// PacketsTable sample (fixed vs variable)
0x0068, // 0x00 — 104 bytes fixed
0x0005, // 0x01 — 5 bytes fixed
-1,     // 0x03 — variable length (read bytes 1-2)
-1,     // 0x0C — variable length

Opcode Dispatch

PacketParser is a singleton with a PacketHandler[256] array. Handlers are delegates with signature (World world, ref StackDataReader reader). Registration happens at construction from PacketHandlerRegistry.GetHandlers(). 

// PacketParser constructor
foreach ((uint id, PacketHandler handler) in PacketHandlerRegistry.GetHandlers())
    AddHandler(id, handler, false);

// Dispatch (inside ParsePackets)
PacketHandler handler = _handlers[opcode];
if (handler != null)
    handler(world, ref reader);   // ← called on main thread

Every handler receives a ref StackDataReader which wraps the packet bytes. The reader provides typed read methods: ReadUInt8(), ReadUInt16BE(), ReadUInt32BE(), ReadASCII(n), ReadUnicode(n), etc. Handlers may also register additional handlers at runtime via PacketParser.Instance.AddHandler() (used by plugins).

Building & Sending Outgoing Packets

All outgoing packets are built using StackDataWriter (a stack-allocated byte writer) and sent via socket.Send(writer.BufferWritten). The pattern is consistent across all packet types: 

// Example: Send_Ping (OutgoingPackets.cs)
public static void Send_Ping(this AsyncNetClient socket, byte idx)
{
    const byte ID = 0x73;
    int length = AsyncNetClient.PacketsTable.GetPacketLength(ID);

    // For fixed-length packets: allocate exact size
    // For variable-length: start with 64 bytes (resizes if needed)
    var writer = new StackDataWriter(length < 0 ? 64 : length);
    writer.WriteUInt8(ID);

    if (length < 0)
        writer.WriteZero(2);   // placeholder for length field

    writer.WriteUInt8(idx);

    if (length < 0)
    {
        // Backfill actual length at bytes 1-2
        writer.Seek(1, SeekOrigin.Begin);
        writer.WriteUInt16BE((ushort)writer.BytesWritten);
    }
    else
    {
        writer.WriteZero(length - writer.BytesWritten); // pad to fixed length
    }

    socket.Send(writer.BufferWritten);
    writer.Dispose();
}
StackDataWriter
StackDataWriter uses stack memory for small packets and rents from ArrayPool for larger ones. Always call writer.Dispose() after sending to return rented buffers. The Send() call copies writer.BufferWritten into the circular send buffer, so it's safe to dispose immediately after.

Encryption

TazUO automatically selects the correct encryption based on client version. EncryptionHelper.CalculateEncryption(clientVersion) derives three 32-bit keys from the version number and returns the appropriate encryption type:

EncryptionTypeWhen usedAlgorithm
NONEServers that disable encryptionPass-through
OLD_BFISHClients < 1.25.35Blowfish with old key schedule
BLOWFISH__1_25_361.25.36 – 2.0.2Blowfish variant
BLOWFISH2.0.3 – modern non-SAStandard Blowfish
BLOWFISH__2_0_3Client 2.0.3 special caseFixed keys: 0x2D13A5FC…
TWOFISH_MD5SA clients (7.0+)Twofish + MD5 key derivation

The encryption type can also be forced via the -encryption N CLI flag, where N maps to the EncryptionType enum value. This is useful when connecting to servers that don't match the standard version-to-encryption mapping.

Login packets use LoginCrypt (a separate XOR-based stream cipher) for the initial handshake, regardless of the game encryption type. After the handshake, EnableCompression() is called on the client to enable Huffman decompression and switch to game encryption.

Thread Safety Rules

ResourceWho may accessHow
World.Mobiles, World.ItemsMain thread onlyPlain Dictionary<uint, T> — no thread safety
World.Player and all GameObject fieldsMain thread onlyNo synchronisation; accessed only in Update/Draw
AsyncNetClient._incomingMessagesReceive task (write) + main thread (read)ConcurrentQueue<byte[]>
AsyncNetClient._sendStreamMain thread (write) + send task (read)lock(_sendStream) on both sides
World._corpsesMain thread (write) + any (read)lock(_corpsesLock); use GetCorpseSnapshot()
AsyncNetClient.SocketAny thread (the singleton reference itself)The static reference is set once; the object is thread-safe for Send() and TryDequeuePacket()
Plugin packet injection
Send() calls Plugin.ProcessSendPacket() before encrypting. Plugins may modify or suppress outgoing packets. If you see unexpected packet suppression, check if a plugin is active via Plugin.Enabled.

Going Forward

Adding a New Incoming Packet Handler

  1. Create Network/PacketHandlers/MyPacket.cs with a static Receive(World world, ref StackDataReader reader) method.
  2. Register it in HandlerRegistry.GetHandlers(): add (0xXX, MyPacket.Receive) to the list.
  3. If the packet is variable-length, add −1 at the opcode index in PacketsTable; otherwise add the fixed byte count.
  4. Read fields from reader in order. The reader automatically tracks position.

Adding a New Outgoing Packet

  1. Add an extension method Send_MyPacket(this AsyncNetClient socket, ...) in OutgoingPackets.cs.
  2. Follow the StackDataWriter pattern: write opcode, optional length placeholder, data fields, then backfill length for variable packets.
  3. Always writer.Dispose() after socket.Send().

Safe Patterns

  • Never call AsyncNetClient.Socket.Send() from the receive or send background tasks — it is designed to be called from the main thread only (it locks the send stream).
  • Never access World state from inside a background task or timer callback.
  • Use World.GetCorpseSnapshot() for any cross-thread corpse access.
  • If you need to trigger UI actions from a packet handler, do it directly — handlers already run on the main thread.
