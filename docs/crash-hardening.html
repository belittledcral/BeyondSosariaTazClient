Crash Hardening — Rendering & Concurrency

Technical reference for the systematic stability sweep committed to TazUO (2026-03-04). Covers what was changed, why it was crashing, and how to write safe code going forward.

What Was Fixed & Why

TazUO was experiencing frequent crashes, most originating in the rendering subsystem. Four distinct root causes were identified and addressed in a single commit:

Phase 1 — Null/Destroyed objects in render loop Phase 2 — Animation frame index overflow Phase 3 — Dictionary mutation during enumeration Phase 4 — Follow-target null (pre-existing guard confirmed)

Root Causes Explained

1. Stale object references in the render list

Every frame, the engine builds a render list — an ordered List<GameObject> of everything that needs to be drawn. Objects are added to this list during the update phase, then iterated and drawn during the draw phase.

Between those two phases (or even during the update), the network thread can receive a packet that destroys a mobile or item. If that happens, the reference in the render list now points to a logically destroyed object. Calling Draw() on such an object reaches into null animation frames, null textures, or de-allocated native resources — resulting in a NullReferenceException or access violation.

2. TNext pointer invalidation during linked-list traversal

Map tiles are stored as intrusive singly-linked lists via the TNext field on GameObject. The traversal pattern obj = obj.TNext reads TNext after the loop body runs. If anything in the loop body — or a concurrent thread — unlinks an object before TNext is read, the iterator jumps to a stale memory location.

3. Silent (byte) overflow on animation frame index

The animation frame index was cast to byte to clamp it to the last frame. A byte can only hold values 0–255. If an animation has more than 256 frames (which is legal in UO data), (byte)(frames.Length - 1) silently wraps — for example, frame 256 becomes 0, frame 300 becomes 44. The next array access then uses this wrong index, causing an out-of-bounds read.

4. Dictionary enumeration during network-thread mutation

World.Mobiles and World.Items are Dictionary-backed collections. The main/render thread iterates them to call Update(), draw health bars, or scan for loot. Meanwhile the network thread can call packet handlers that add or remove entries. The .NET Dictionary enumerator detects this and throws InvalidOperationException: Collection was modified.

Phase 1 — Rendering: Pipeline Overview

Each game frame executes a two-stage pipeline:

┌─────────────────────────────────────────────────────────────────┐ │ GAME LOOP FRAME │ ├──────────────────────────┬──────────────────────────────────────┤ │ UPDATE PHASE │ DRAW PHASE │ │ │ │ │ GameScene.Update() │ GameScene.Draw() │ │ └─ World.Update() │ └─ DrawRenderList(statics) │ │ ├─ Mobiles │ └─ DrawRenderList(animations) │ │ ├─ Items │ └─ DrawRenderList(effects) │ │ └─ Effects │ └─ DrawRenderList(transparent) │ │ │ │ │ SortAndAddToRenderList()│ │ │ └─ AddTileToRenderList│ │ │ (fills the 4 │ │ │ render lists) │ │ └──────────────────────────┴──────────────────────────────────────┘ Network thread (async) can mutate Mobiles / Items at ANY time ──────►

The four render lists are:

  • _renderListStatics — land tiles, static objects, most items
  • _renderListAnimations — mobiles, corpses
  • _renderListEffects — particle/spell effects
  • _renderListTransparentObjects — foliage, trees with alpha

DrawRenderList — Null/Destroyed Guard

GameScene.cs · DrawRenderList()

What it does

DrawRenderList iterates one of the four render lists and calls obj.Draw(batcher, x, y, depth) for every object whose Z-coordinate is within the current ground ceiling (_maxGroundZ).

GameScene.cs — DrawRenderList()
                    Profiler.EnterContext("Draw");
                    if (obj == null || obj.IsDestroyed)
                    {
                        Profiler.ExitContext("Draw");
                        continue;
                    }
                    if (
                        obj.Draw(batcher, obj.RealScreenPosition.X, obj.RealScreenPosition.Y, depth)
                    )
                    {
                        ++done;
                    }
                    Profiler.ExitContext("Draw");
Why the check is inside the Profiler context
The profiler context was entered before the guard. Exiting it explicitly before continue keeps profiler call pairs balanced, preventing a context stack corruption that would skew performance metrics.

Why this crashes without the guard

Individual Draw() overrides in subclasses do have their own early-return guards (if (IsDestroyed) return false), but these guards execute inside virtual dispatch. Virtual dispatch itself is safe — the issue is that the object may have had its native frame data freed between when it was added to the render list and when the draw loop reaches it. The null check at the call site prevents us from ever entering that corrupted state.

TNext Linked-List Safety

GameSceneDrawingSorting.cs

Data structure

Each tile cell in the map holds a singly-linked chain of GameObject instances stacked on that cell (land, then statics, then items, then mobiles). Each node exposes its successor via the TNext property.

chunk[x,y] ──► Land ──► Static ──► Static ──► Item ──► Mobile ──► null │ │ │ │ │ TNext TNext TNext TNext TNext

The hazard

The classic for (obj = head; obj != null; obj = obj.TNext) reads obj.TNext after the loop body runs. If anything during the loop body — or a concurrent network packet — removes obj from the chain, TNext may be zeroed out or point to freed memory before the loop reads it.

The fix — pre-capture pattern

Capture TNext as the very first thing in the loop body, before any code that could trigger removal. Use it as the for-loop update expression so that continue and end-of-body both advance through the pre-captured pointer:

Loop pattern — before / after
// ── BEFORE (unsafe) ──────────────────────────────────
for (GameObject obj = chunk.GetHeadObject(x, y); obj != null; obj = obj.TNext)
{
    // ... body. TNext is read AFTER the body by the for-update.
    // If body triggers removal, TNext may be stale by then.
}

// ── AFTER (safe) ─────────────────────────────────────
for (GameObject obj = chunk.GetHeadObject(x, y), _tnext = null; obj != null; obj = _tnext)
{
    _tnext = obj.TNext;   // ← captured FIRST, before any body code
    // ... rest of body. continue / break both work correctly.
}

How the for-loop trick works

In a for (init; condition; update) loop, the update expression runs after the body — including after any continue statement. By using _tnext as the update expression and setting _tnext = obj.TNext at the top of the body:

  • continue → skips to the update → obj = _tnext (already captured) ✓
  • Normal fall-through → update → obj = _tnext ✓
  • break → exits the loop entirely ✓
  • return → exits the function entirely ✓

No loop body code needed to change. This pattern was applied to all four loop sites:

Variable Location Purpose of loop
_tnext GetMaxLandZ — tile at player position Determine max visible Z / roof culling
_tnext2 GetMaxLandZ — tile at player+1 diagonal Roof culling look-ahead one tile NE
_tnext3 ApplyFoliageTransparency Mark tree tiles with current foliage index
_tnext4 AddTileToRenderList Classify and enqueue tile objects into render lists

Classification logic per object type

Each object in the TNext chain is pattern-matched via switch (obj) and routed to the appropriate render list:

Type Target list Key skip conditions
Land _renderListStatics Above maxZ → return false; out of Y viewport → skip
Static _renderListStatics or _renderListTransparentObjects Internal flag; not visible at current season; alpha-culled; foliage hidden; out of viewport
Multi _renderListStatics or _renderListTransparentObjects Same as Static plus movable-multi foliage rules
Mobile _renderListAnimations Exceeds character height ceiling; alpha-culled; out of viewport; underground (HasSurfaceOverhead)
Item _renderListAnimations (corpse) or _renderListStatics Internal; seasonal foliage; alpha-culled; foliage-to-stump; out of viewport
GameEffect _renderListEffects Alpha-culled; out of viewport
Tree shadow hack
Trees and rocks are placed in _renderListTransparentObjects instead of _renderListStatics when shadows are enabled. This avoids Z-fighting between the object and its shadow because transparent objects are drawn in a separate back-to-front pass.

Return value semantics

The function returns false normally and true only for a special case: a tall Static or Multi that overhangs the Z ceiling. Returning true tells the caller (SortAndAddToRenderList) that partial rendering of the overhanging tile is valid — this prevents the roof from completely disappearing when the player stands near its edge.

Phase 2 — Animations: System Overview

TazUO renders character and corpse animations by indexing into a Span<SpriteInfo> returned by: 

animations.GetAnimationFrames(id, action, dir, out _, out _)

The span points into native memory loaded from UO's anim*.mul / UOP files. AnimIndex (a byte field on every Entity) stores the current frame. On each ProcessAnimation() call, AnimIndex is incremented or decremented, then stored back.

Server-scripted animation modes

When AnimationFromServer is true the server controls the animation, supplying:

  • _animationInterval — multiplier applied to the base frame delay
  • AnimationFrameCount — overrides how many frames are in the clip (0 = use actual count)
  • _isAnimationForwardDirection — play forward or backward
  • _animationRepeateMode — 0 = loop forever, N = play N times then stop

{
    frameIndex = frames.Length - 1;   // clamp: hold last frame
}
AnimIndex = (byte)(frameIndex % frames.Length);

The ExecuteAnimation flag controls whether to advance the frame this tick. A corpse that has reached its final frame stops there rather than looping.

The Frame Index Overflow Bug

The bug

Both files previously used a (byte) cast to assign the last-frame index:

Mobile.cs — server-driven backward wrap
frameIndex = (byte)(frames.Length - 1);
frameIndex = frames.Length - 1;
Item.cs — corpse end-of-clip clamp (declaration also changed)
byte frameIndex = (byte)(AnimIndex + (ExecuteAnimation ? 1 : 0));
int frameIndex = AnimIndex + (ExecuteAnimation ? 1 : 0);

  // later …
frameIndex = (byte)(frames.Length - 1);
frameIndex = frames.Length - 1;

Why it overflows

A byte holds exactly 8 bits: values 0–255. The expression (byte)(frames.Length - 1) is computed in int first, then truncated to the low 8 bits. For any animation with more than 256 frames:

frames.Length(byte)(frames.Length - 1)Should be
256255 ✓255
2570 ✗256
30043 ✗299
512255 ✓ (coincidence)511

The wrong index then reaches frames[frameIndex], accessing either the wrong sprite or — in severe cases — memory outside the span's bounds.

Fix
In Mobile.cs frameIndex was already declared int; removing the cast is sufficient.

In Item.cs frameIndex was declared byte. Changing the declaration to int is the correct fix — it allows the variable to hold any valid array index. The final AnimIndex = (byte)(frameIndex % frames.Length) assignment remains correct because after the modulo the result always fits in a byte (AnimIndex itself is a byte field on Entity).

Phase 3 — Concurrency: Threading Model

TazUO runs two threads that share game state:

┌──────────────────────────────────────────────────────────────────┐ │ MAIN / RENDER THREAD │ │ │ │ Update() → Draw() (runs every frame, ~60 fps) │ │ • Iterates Mobiles, Items │ │ • Reads world state to render health bars, loot, etc. │ └─────────────────────────┬────────────────────────────────────────┘ │ shared: World.Mobiles, World.Items ┌─────────────────────────▼────────────────────────────────────────┐ │ NETWORK THREAD │ │ │ │ AsyncSocketWrapper.ReadAsync() → PacketHandlers │ │ • AddMobile(), RemoveMobile() │ │ • AddItem(), RemoveItem() │ │ • Fires at any time, unrelated to the game loop │ └──────────────────────────────────────────────────────────────────┘
Dictionary is not thread-safe
Dictionary<K,V> in .NET is explicitly documented as not thread-safe for concurrent reads and writes. The enumerator holds a version counter; if the dictionary is mutated while the enumerator is active, it throws InvalidOperationException. This is not a rare edge case — on a busy server it can happen multiple times per second.

World.Update — Mobiles & Items Loops

World.cs · Update()

Mobiles update loop

Profiler.EnterContext("MobilesUpdate");
foreach (Mobile mob in Mobiles.Values.ToList())   // snapshot
{
    mob.Update();

    {
        RemoveMobile(mob);    // modifies Mobiles — safe because we iterate a snapshot
    }

    if (mob.IsDestroyed)
    {
        _toRemove.Add(mob.Serial);
    }
    else
    {
        // update war-map overlay …
    }
}
Profiler.ExitContext("MobilesUpdate");

// deferred removal — avoids issues even without the snapshot
foreach (uint serial in _toRemove)
    Mobiles.Remove(serial);
_toRemove.Clear();

Deferred removal pattern

Even within the same thread, RemoveMobile internally removes the entry from the dictionary. With the .ToList() snapshot the enumerator is already decoupled from the live dictionary, so direct removal during the loop is safe. The _toRemove list handles the separate case of objects that set IsDestroyed = true without being caught by the distance check.

do_delete throttle

Range cleanup does not run every frame. _timeToDelete gates it to run at most every 50 ms. This prevents O(n) distance checks from dominating CPU time on frames where nothing else is happening.

AutoLootManager — Scavenger Loop

AutoLootManager.cs · OnPositionChanged()

How the scavenger works

The scavenger is a feature that automatically picks up ground items (not corpses) when the player walks near them. It fires from the PositionChanged event and is rate-limited to run at most once every 500 ms. 

private void OnPositionChanged(object sender, PositionChangedArgs e)
{
    if (!_loaded) return;

    if (ProfileManager.CurrentProfile.EnableScavenger && Time.Ticks >= _nextScavengerScan)
    {
        _nextScavengerScan = Time.Ticks + 500;
        foreach (Item item in _world.Items.Values.ToList())   // snapshot
            if (item != null && item.OnGround && !item.IsLocked && !item.IsCorpse && item.Distance < 3)
                CheckAndLoot(item);
    }

    if (IsEnabled)
        foreach (Item corpse in _world.GetCorpseSnapshot())   // already safe
            CheckCorpse(corpse);
}

Why corpse iteration is already safe

_world.GetCorpseSnapshot() returns a pre-made thread-safe array copy of the corpse list (guarded by _corpsesLock). This pattern is correct and was not changed. Only the Items.Values scavenger loop needed the .ToList() addition.

Going Forward — Coding Guidelines

These are the rules that prevent the bugs fixed in this commit from reappearing.

Safe Patterns

Iterating Mobiles or Items

Always snapshot before iterating shared collections
// ✓ Safe — snapshot decouples iterator from live dictionary
foreach (Mobile mob in _world.Mobiles.Values.ToList())
{ ... }

// ✓ Also safe — GetCorpseSnapshot() returns a locked array copy
foreach (Item corpse in _world.GetCorpseSnapshot())
{ ... }

Iterating TNext linked lists

Always pre-capture TNext as the first statement in the loop body
// ✓ Safe — TNext captured before any code that could invalidate it
for (GameObject obj = startNode, _tnext = null; obj != null; obj = _tnext)
{
    _tnext = obj.TNext;   // ← must be FIRST
    // ... safe to use continue, break, return anywhere below
}

Calling Draw() on game objects

Guard null and IsDestroyed before calling Draw()
// ✓ Safe
if (obj == null || obj.IsDestroyed)
    continue;
obj.Draw(batcher, x, y, depth);

Animation frame indices

Use int for frame indices; never (byte)-cast an array length expression
// ✓ Safe — int can hold any frame count
int frameIndex = AnimIndex + delta;

if (frameIndex >= frames.Length)
    frameIndex = frames.Length - 1;           // no cast needed

AnimIndex = (byte)(frameIndex % frames.Length); // cast AFTER modulo is safe

Anti-Patterns to Avoid

Never use (byte) to clamp an array length or index
// ✗ WRONG — overflows silently for frames.Length > 256
frameIndex = (byte)(frames.Length - 1);
Never iterate Mobiles/Items without ToList() (or equivalent snapshot)
// ✗ WRONG — throws InvalidOperationException if network thread mutates the dictionary
foreach (Mobile mob in _world.Mobiles.Values)
{ ... }
Never read obj.TNext after the loop body without pre-capturing it
// ✗ WRONG — TNext may be stale if body (or another thread) unlinks obj
for (GameObject obj = head; obj != null; obj = obj.TNext)
{
    // ... body that could trigger removal ...
    // obj.TNext is read HERE by the for-update, AFTER the body
}
Never call Draw() without a destroyed/null check when the object came from a shared list
// ✗ WRONG — render list can contain stale entries by draw time
foreach (GameObject obj in renderList)
{
    obj.Draw(batcher, x, y, depth); // obj may be destroyed between update and draw
}
Performance note on .ToList()
.ToList() allocates a new List<T> on the heap every call. For the Mobiles and Items loops (which run every frame) this generates GC pressure. If this becomes a measurable problem, replace with a pre-allocated snapshot buffer or a ReaderWriterLockSlim around the dictionary. For manager-level loops (HealthLines, AutoLoot) that run much less frequently, .ToList() is fine.
