UI Gumps: Key Windows

Session 08 — The major game windows, server vs. client gumps, and state persistence

Overview

The Game/UI/Gumps/ directory contains ~90 files — every major gameplay window. Some gumps are client-created (spawned by game code in response to player actions), while others are server-created (generic gumps sent as packets by the UO server). Most TazUO gumps are client-created.

The class hierarchy for gumps adds several optional layers on top of the base Gump:

Gump — base (position, lock, alpha, save/restore) ├── AnchorableGump — can be snap-anchored to adjacent gumps (health bars) │ ├── BaseHealthBarGump — any HP-tracking window │ │ ├── HealthBarGump — classic UO art health bar │ │ └── ModernHealthBar │ └── ModernPaperdoll — TazUO flat paperdoll ├── ResizableGump — can be resized by dragging an edge/corner │ ├── GridContainer — TazUO grid inventory view │ └── ResizableJournal — resizable journal (TazUO) ├── ScalableGump — can be scaled (zoomed) │ └── ScalableTextContainerGump │ └── PaperDollGump — classic UO paperdoll ├── TextContainerGump — manages a list of gumps from text packets │ └── ContainerGump — classic UO container view └── StatusGumpBase — player status bar (HP/Mana/Stam/Stats) ├── StatusGumpModern — TazUO modern status bar └── StatusGumpOld — classic UO status bar

PaperDollGump & ModernPaperdoll

ModernPaperdoll (TazUO) TazUO

ModernPaperdoll : AnchorableGump replaces the classic art with a flat modern design. It loads modern-paperdollgump.png from embedded resources via PNGLoader instead of using UO gumpart IDs.

FeatureDetail

Both paperdolls register as GumpType.PaperDoll. Only one can exist at a time. The decision of which to show is made by Profile.UseModernPaperdoll.

ContainerGump (Classic)

ContainerGump : TextContainerGump displays a container's contents using the UO gumpart art as the background. Items are draggable ItemGump controls positioned by their in-container coordinates.

Backpack Style

When the container is the player's backpack and the client version ≥ 7.0.53.1, the graphic ID is replaced with a style-specific one from Profile.BackpackStyle:

Grid Toggle

A "Switch to Grid" button appears when grid containers are enabled (Profile.UseGridLayout). Clicking it opens a GridContainer for the same container serial and closes the classic view. The reverse button in GridContainer reopens the classic view.

Corpse Animation

When LocalSerial is a corpse, ContainerGump animates the eye gump (graphic 0x0045) by cycling between frames on a timer — the classic "blinking eye" on corpse bags.

GridContainer TazUO

Layout Math

// Number of columns that fit in the current width
int columns = (Width - _borderWidth * 2) / (GridItemSize + X_SPACING);

// Each cell rendered at:
int cellX = _borderWidth + (col * (GridItemSize + X_SPACING));
int cellY = TOP_BAR_HEIGHT + LABEL_HEIGHT + (row * (GridItemSize + Y_SPACING));

GridContainerSaveData

The GridContainerSaveData manager persists per-container layout settings (column count, preferred size, border style) keyed by container serial. This means each bag remembers its own grid configuration independently.

HealthBarGump

BaseHealthBarGump : AnchorableGump is the parent for all mobile health bar windows. It provides serial binding, mobile status request, and the anchor system.

AnchorableGump

Health bars can be snapped together into groups via the AnchorManager. Dragging one bar drags the entire anchored group. This is how party health bars are typically organized into a stacked block.

NewTargetSystem Integration

When Profile.UseNewTargetSystem is true, the gump bound to TargetManager.NewTargetSystemSerial is highlighted with a colored border. SetNewMobile(uint serial) can be called to re-bind an existing health bar to a different mobile, clearing children and rebuilding instead of creating a new gump.

IsLastAttackBar

The static LastAttackBar tracks which health bar represents the last attacked target. Only one bar at a time can be the "last attack" bar. GameScene sets this when attack packets are received.

StatusGump

StatusGumpBase : Gump is the player status bar showing HP, Mana, Stamina, and character attributes. TazUO provides two concrete implementations chosen by Profile.StatusGumpType:

ClassStyle
StatusGumpOldClassic UO status gump art (small, fixed layout)
StatusGumpModernTazUO modern status bar (resizable, extended stats)

Mutual Exclusion

If Profile.StatusGumpBarMutuallyExclusive is true, opening the StatusGump automatically disposes any existing HealthBarGump for the player (and vice versa), preventing both from showing the same HP information simultaneously.

Buff Toggle

The BuffIcon button on the status gump opens either ImprovedBuffGump or the classic BuffGump depending on Profile.UseImprovedBuffBar. If the gump is already open, it is brought to front via SetInScreen() + BringOnTop() rather than duplicated.

Stat Lockers

Three GumpPic controls (lock up, lock down, lock locked) let players cycle the STR/DEX/INT stat lock state. Clicking cycles through the three images and sends an OutgoingPacket to the server.

JournalGump

JournalGump : Gump is the classic UO journal/chat log. It uses an ExpandableScroll background (a UO art 9-slice that stretches vertically) and a RenderedTextList for the scrollable text content.

Dark Mode

A checkbox toggles dark mode by applying hue 903 to the scroll graphic, darkening the background art. The state is saved in Profile.JournalDarkMode.

Message Type Filters

Four Checkbox controls let players filter the log by MessageType category (System, Regular, Emote, Label). The filter state is saved per-session.

ResizableJournal (TazUO) TazUO

ResizableJournal : ResizableGump replaces the classic journal with a freely resizable window. It adds a tab system for multiple named journal channels (each backed by a separate entry list), timestamp display, and copy-to-clipboard functionality.

Skill Gumps

TazUO provides two skills windows selectable via Profile.UseAdvancedSkillMenu:

StandardSkillsGump

Classic UO skills list: a scrollable column of skill name + value + lock state. Uses the original UO art.

SkillGumpAdvanced (TazUO) TazUO

A modern redesign with grouping, sorting, progress bars per skill, and category headers. Reads from World.Player.Skills[] and subscribes to EventSink.SkillsChanged to update in real time as skill values change.

SkillButtonGump

A floating "Use Skill" shortcut button, similar to UseSpellButtonGump. Players drag a skill name from the skills window to create one. Persists as GumpType.SkillButton.

SkillProgressBar (TazUO) TazUO

Shows a visual progress bar for skill gain towards the next 0.1 increment. Appears briefly when a skill gains value.

WorldMapGump

WorldMapGump renders the Ultima Online world at a configurable zoom level using map data from MapLoader. TazUO extends it with:

  • Markers — player-placed markers with labels, managed by MarkersManagerGump
  • Party display — party member locations shown as colored dots
    • Goto — LocationGoGump dialog for jumping to specific coordinates
  • Guild map integration — shows server-sent guild marks
  • MapWebServer — live HTTP tile server for browser-based mapping tools

Server-Created Gumps

Some gumps are created by the server via a Generic Gump packet (opcode 0xB0). The packet contains a GumpID, Gump layout string, and text array. TazUO parses this into controls:

Server → Generic Gump Packet (0xB0) PacketHandlers.OnOpenGump() └── Gump layout string parsed: "page 0" → set ActivePage "resizepic X Y G W H" → new ResizePic "button X Y G1 G2 flag ID" → new Button "text X Y HUE IDX" → new Label (from lines[IDX]) "htmlgump X Y W H IDX BKND SCRL" → new HtmlControl "checkertrans X Y W H" → new CheckerTrans "croppedtext ..." → new CroppedText (and ~40 more keywords) └── Add to Gumps as Gump with IsFromServer=true

Server gumps use ServerSerial as their key. When the player responds (clicks a button), a GenericGumpResponse packet is sent with the button ID. Server gumps cannot be saved across sessions (they live only until the server closes them).

Server gumps vs. client gumps Never confuse the two. Server gumps are destroyed when the server sends a "Close Gump" packet or when the player right-clicks. Client gumps are created/destroyed entirely by client code and are independent of server state.

Gump State Persistence

On logout (scene change from GameScene → LoginScene), GameScene.Save() iterates all gumps and writes those with CanBeSaved = true to {ProfilePath}/layout.xml:

GameScene.Save() └── foreach Gump in UIManager.Gumps ├── if not CanBeSaved → skip └── gump.Save(XmlTextWriter) ├── type (GumpType enum int) ├── x, y (position) ├── serial (LocalSerial) ├── serverSerial ├── isLocked ├── alphaOffset └── gump-specific attributes (override Save())

On login (GameScene loading), the XML is read and each gump is re-created by type:

GameScene.Load() └── Deserialize layout.xml └── foreach element in xml ├── read type → GumpType enum ├── switch(type): │ case GumpType.PaperDoll → new PaperDollGump(world, serial, ...) │ case GumpType.Container → reopen container if item still exists │ case GumpType.HealthBar → new HealthBarGump(world, serial) │ case GumpType.GridContainer → new GridContainer(world, serial, ...) │ ... └── gump.Restore(XmlElement) ├── IsLocked = ... └── Alpha += alphaOffset

Custom Save/Restore

Gumps that store additional state override Save() and Restore():

public override void Save(XmlTextWriter writer)
{
    base.Save(writer);   // always call base first (writes type, x, y, serial, lock, alpha)
    writer.WriteAttributeString("myCustomSetting", _myValue.ToString());
}

public override void Restore(XmlElement xml)
{
    base.Restore(xml);   // restores lock, alpha
    if (int.TryParse(xml.GetAttribute("myCustomSetting"), out int v))
        _myValue = v;
}

Container Auto-Reopen

Container gumps (classic and grid) are only re-created on login if the container item still exists in the world. The XML stores the container serial; the restore code calls World.Items.Get(serial) — if it returns null, the gump is skipped.

ModernOptionsGump TazUO

ModernOptionsGump.cs is the game's settings screen, opened via the escape menu or a keybind. It uses a left-nav / right-content layout where each nav button corresponds to a PAGE enum value. All page content is built at construction time into a shared options list of SettingsOption objects; visibility is driven by the active page integer.

Page enum

The PAGE enum (bottom of the file) maps every tab to an integer. Adding a new tab requires:

  1. A new PAGE enum value
  2. A CategoryButton(…) call in Build()
  3. A language string in Language.cs
  4. A Build<Page>() method that adds SettingsOption objects with the matching page int

Plugins page TazUO

The Plugins tab (PAGE.Plugins) scans Data/Plugins/*.dll at options-open time and renders a checkbox per DLL. Toggling a checkbox immediately rewrites Settings.GlobalSettings.Plugins (a string[] in settings.json) and calls Settings.GlobalSettings.Save(). Plugin loading happens at startup only, so a restart notice is shown at the top of the page.
