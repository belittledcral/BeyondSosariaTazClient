Architecture & Entry Points

How TazUO boots, what the game loop looks like, and the core infrastructure every system builds on: the scene stack, profile config, and input pipeline.

What is TazUO?

TazUO is a feature-complete fork of ClassicUO, the open-source reimplementation of the Ultima Online 2D client. The original ClassicUO faithfully reproduced the legacy client; TazUO extends it with community-requested quality-of-life features, a Python scripting engine (LegionScripting / IronPython), grid-based inventory management, modern ImGui overlays, and deep per-character customisation.

Architecturally the two codebases are close — the same UO file-format parsers, the same FNA/XNA rendering pipeline, the same network packet structure. The significant additions in TazUO live in:

AreaTazUO additionsLocation
ScriptingIronPython 3.4.2 runtime with full C# API bridgeLegionScripting/
InventoryGrid containers, property-based highlighting, auto-lootGame/UI/Gumps/GridContainer.cs
HUDBuff bars, cooldown bars, overhead health lines, name overheadGame/Managers/
ConfigSQLite-backed settings + 500+ per-profile JSON propertiesConfiguration/
ImGuiModern overlay UI coexisting with legacy UO gumpsGame/UI/ImGuiControls/
Data persistenceWorldDataManager for per-map saved dataGame/Managers/WorldDataManager.cs
Dependency on FNA
TazUO uses FNA (an XNA reimplementation) rather than MonoGame. This means Microsoft.Xna.Framework types (Game, GameTime, Texture2D, GraphicsDevice, etc.) are provided by FNA and compiled into the output directory as native libraries. The game loop contract (Initialize, LoadContent, Update, Draw) is the same as classic XNA.

Program Startup

The entry point lives in src/ClassicUO.Client/Main.cs inside the Bootstrap static class. The Main() method calls Boot(), which is also reachable via an unmanaged [UnmanagedCallersOnly] export named Initialize — used when the client is loaded as a plugin DLL by a launcher.

Bootstrap.Main(args) └─ Bootstrap.Boot(pluginHost, args) │ ├─ CopyRequiredLibs() // copy x64/lib64/osx native DLLs next to exe ├─ CultureInfo → InvariantCulture // ensure consistent number formatting ├─ Language.Load() // load localised strings ├─ Log.Start(LogTypes.All) // initialise logging ├─ CUOEnviroment.GameThread = Thread.CurrentThread │ ├─ AppDomain.UnhandledException // crash handler → HTML crash log │ ├─ ReadSettingsFromArgs(args) // first pass: -settings path override ├─ Settings.GlobalSettings = ConfigurationResolver.Load(...) ├─ ReadSettingsFromArgs(args) // second pass: override individual fields │ ├─ Validate UO directory & client version │ (flags INVALID_UO_DIRECTORY | INVALID_UO_VERSION) │ ├─ Set FNA environment variables │ FNA3D_D3D11_FORCE_BITBLT=1 │ FNA3D_FORCE_DRIVER = OpenGL | Vulkan (per settings) │ └─ Client.Run(pluginHost) // enter the game

Native Library Copy

Before anything else, CopyRequiredLibs() copies the platform-appropriate native DLLs (SDL3, FNA3D, FAudio, etc.) from a versioned subdirectory to the executable directory, so FNA can load them by name. The source directories are:

  • external/x64/ — Windows x64
  • external/lib64/ — Linux x64
  • external/osx/ or external/osx-arm/ — macOS

Files are only copied if missing or if the sizes differ, so repeated launches are fast.

Command-Line Arguments

ReadSettingsFromArgs() parses -key value pairs and directly mutates Settings.GlobalSettings. It is called twice — once before loading the JSON (to allow -settings <path> to redirect the config file) and once after (to let CLI arguments override whatever the JSON contained).

The Client.Game Singleton

src/ClassicUO.Client/Client.cs contains two types that act as the global hub of the entire client: 

// The static façade — every system reaches through this
internal static class Client
{
    public static GameController Game { get; private set; }
    public static SQLSettingsManager Settings { get; private set; }

    public static void Run(IPluginHost pluginHost)
    {
        Settings = new SQLSettingsManager();
        using (Game = new GameController(pluginHost))
        {
            Game.Run();   // blocks until window closes
        }
    }
}

// The asset + world container, accessible via Client.Game.UO
sealed class UltimaOnline
{
    public World              World      { get; private set; }
    public GameCursor         GameCursor  { get; private set; }
    public Renderer.Animations.Animations Animations { get; ... }
    public Renderer.Arts.Art             Arts       { get; ... }
    public Renderer.Gumps.Gump           Gumps      { get; ... }
    public UOFileManager      FileManager { get; private set; }
    // ... Texmaps, Lights, MultiMaps, Sounds
}

UltimaOnline.Load() is called once from GameController.LoadContent(). It validates the UO client path and version, maps the version to a set of ClientFlags (T2A, RE, LBR, AOS, SE, SA expansions), loads all UO files via UOFileManager, then creates the GPU-side renderer objects and the World.

Access pattern
From anywhere in the client, the canonical access paths are:
  • Client.Game — the FNA game/window/graphics device
  • Client.Game.UO.World — all game objects
  • Client.Game.UO.Arts — art texture cache
  • Client.Game.Scene — the currently active scene
  • ProfileManager.CurrentProfile — per-character settings

The Game Loop

GameController extends FNA's Microsoft.Xna.Framework.Game. FNA drives the loop at the configured framerate (default 60 FPS, configurable up to Constants.MAX_FPS). The key overrides:

FNA runtime calls per frame: ────────────────────────────────────────────────── GameController.Update(GameTime gt) │ ├─ Time.Ticks = (uint)gt.TotalGameTime.TotalMilliseconds ├─ Time.Delta = (float)gt.ElapsedGameTime.TotalSeconds │ ├─ Process SDL events (keyboard, mouse, window resize…) │ → Mouse.ButtonPress / ButtonRelease │ → Keyboard events → UIManager routing │ ├─ Scene?.Update(Time.Ticks) │ └─ [if GameScene] World.Update() │ UIManager.Update() │ Managers update… │ └─ Plugin callbacks (if plugins loaded) GameController.Draw(GameTime gt) │ ├─ [suppress if minimised or no scene] │ ├─ Scene?.Draw(_uoSpriteBatch) │ └─ [if GameScene] │ GameScene.Draw() │ ├─ Render to _worldRenderTarget │ ├─ Apply lighting to _lightRenderTarget │ └─ Composite + post-process to backbuffer │ ├─ UIManager.Draw(_uoSpriteBatch) // gumps on top ├─ ImGui.Render() // ImGui overlay └─ GraphicsDevice.Present() ──────────────────────────────────────────────────

Fixed vs Variable Timestep

SetRefreshRate(fps) computes a FrameDelay array with two entries: the normal frame delay and a halved value used for background frames (when the window has no focus). The profiler tracks elapsed time and skips Update/Draw if called too early, implementing a manual fixed-timestep on top of FNA's own timing. 

// From GameController.SetRefreshRate()
FrameDelay[0] = FrameDelay[1] = (uint)frameDelay;   // foreground
FrameDelay[1] = FrameDelay[1] >> 1;                  // background (half speed)
_intervalFixedUpdate[0] = frameDelay;
_intervalFixedUpdate[1] = 217;                        // 5 FPS minimum

Scene Switching

GameController.SetScene(scene) disposes the current scene and loads the new one: 

public void SetScene(Scene scene)
{
    Scene?.Dispose();    // tear down old scene's managers, gumps
    Scene = scene;
    Scene?.Load();       // initialise new scene
    drawScene = Scene != null && Scene.IsLoaded;
}

Time.Ticks — The Global Timer

src/ClassicUO.Client/Time.cs is a two-field static class that every system uses for timing: 

internal static class Time
{
    public static uint  Ticks;   // milliseconds since game started
    public static float Delta;   // elapsed seconds this frame
}

Time.Ticks is updated at the top of every Update() call from GameTime.TotalGameTime.TotalMilliseconds cast to uint. This gives roughly 49 days before wrapping — long enough for any play session.

Common usage pattern
Systems schedule future actions by storing a deadline and comparing against Time.Ticks. Always use > not == to avoid missing the exact tick: 
// Correct — fires as soon as the deadline passes
if (Time.Ticks > _nextActionTime)
{
    DoAction();
    _nextActionTime = Time.Ticks + 500;   // schedule next in 500 ms
}

// Wrong — may never fire if a frame is skipped
if (Time.Ticks == _nextActionTime) { ... }

Scene System

A Scene is the top-level game state. There is exactly one active scene at a time. Each scene has a Load(), Update(uint ticks), Draw(), and Dispose() lifecycle. The three concrete scenes are:

Scene classWhen activeResponsibility
LoginScene Before login Server selection, login UI, initial network handshake
CharacterCreationScene After login, creating a new character Character appearance editor, skill/stat selection
GameScene In-game World rendering, all game managers, player input, UI
Program launch │ └─ [LoginScene] │ (select server, enter credentials, authenticate) │ ├─ [New character] → CharacterCreationScene │ │ │ (create character, confirm) │ │ └─ [Enter world] ──────────┘ │ └─ [GameScene] ← stays active until logout/disconnect │ └─ [Logout] → LoginScene (recycled)

GameScene Initialisation

GameScene is constructed with the current World instance and immediately applies profile settings to the camera and viewport: 

public GameScene(World world)
{
    _world = world;
    _autoUnequipActionManager = new AutoUnequipActionManager(_world);

    Camera.Zoom = ProfileManager.CurrentProfile.DefaultScale;
    Client.Game.SetScale(ProfileManager.CurrentProfile.RenderScale);
    _world.ClientViewRange = ProfileManager.CurrentProfile.ViewRange;
    Camera.Bounds.X = ProfileManager.CurrentProfile.GameWindowPosition.X;
    Camera.Bounds.Width = ProfileManager.CurrentProfile.GameWindowSize.X;
    // ... etc
}

Serialisation context
Profile uses ProfileJsonContext, a source-generated JsonSerializerContext with a custom snake_case naming policy. Any new property you add to Profile must be a supported JSON-serialisable type. Complex custom types need to be added to the context. The context is declared in the same file as Profile via [JsonSerializable(typeof(Profile))].

Saving the Profile

Profiles are saved automatically at key moments (logout, options dialog close) via ConfigurationResolver.Save(profile, path, ProfileJsonContext.DefaultToUse.Profile). There is no auto-save timer — changes that aren't explicitly saved will be lost on crash.

Input Handling

Input in TazUO flows through SDL3 events, processed in the GameController.Update() SDL event loop, and then distributed to the static Mouse and Keyboard classes before being routed to UIManager.

SDL3 event queue (polled in GameController.Update) │ ├─ SDL_EVENT_MOUSE_BUTTON_DOWN/UP │ └─ Mouse.ButtonPress(type) / Mouse.ButtonRelease(type) │ → SDL_CaptureMouse(true/false) │ ├─ SDL_EVENT_MOUSE_MOTION │ └─ Mouse.Position = new Point(x, y) │ → UIManager.OnMouseMoving(x, y) — hover / drag │ ├─ SDL_EVENT_MOUSE_WHEEL │ └─ Mouse.WheelScrolled += delta │ → UIManager.OnMouseWheel │ ├─ SDL_EVENT_KEY_DOWN / KEY_UP │ └─ Keyboard.Alt / Shift / Ctrl updated │ Keyboard.KeyDownEvent?.Invoke(keyString) │ → UIManager.OnKeyboardDown │ → MacroManager.CheckHotkeys │ └─ SDL_EVENT_TEXT_INPUT └─ UIManager.OnTextInput(text)

Mouse

Mouse (Input/Mouse.cs) is a static class with button state properties for each button type and the current cursor position: 

public static Point  Position        // current cursor, updated every motion event
public static bool   LButtonPressed  // left button held
public static bool   RButtonPressed  // right button held
public static bool   MButtonPressed  // middle button held
public static Point  LClickPosition  // where left was pressed (for drag detection)
public const  int    MOUSE_DELAY_DOUBLE_CLICK = 350  // ms window for double click

Keyboard

Keyboard (Input/Keyboard.cs) normalises key combinations into canonical strings like "CTRL+SHIFT+SDLK_A" for macro and hotkey comparison: 

public static bool   Alt   // modifier state, updated per event
public static bool   Shift
public static bool   Ctrl

// event fired for every key-down (includes modifier state in the string)
public static event Action<string> KeyDownEvent;

// normalise "ctrl+a" → "CTRL+SDLK_A" for consistent hotkey comparison
public static string NormalizeKeyString(string input) { ... }

Input Priority Chain

The UIManager processes input in Z-order from top to bottom of the gump stack. An ImGui window sitting on top of the UO gumps has the highest priority; a focused TextBox control will consume all keyboard input before it reaches MacroManager. This means you must never assume input events reach macro handlers while text input is focused.

Going Forward

Safe Patterns

PatternGuidance
Accessing world objects Always via Client.Game.UO.World. Never cache a World reference in a field across scenes — the World is recreated on login.
Timing / throttling Use Time.Ticks + N as a deadline and Time.Ticks > deadline to check. Avoid DateTime.Now for in-game timing.
Adding profile options Add a public property to Profile with a default value. If the type is complex (not a primitive or Point), add it to the JSON context.
Scene transitions Call Client.Game.SetScene(new YourScene()) from the game thread. Never call from the network thread.
Input handling Subscribe to Keyboard.KeyDownEvent or handle in a gump's OnKeyboardDown() override, not by polling SDL directly.

Anti-Patterns to Avoid

  • Static fields that hold World objects: If the player logs out and back in, the World is a new instance. Static caches will hold stale (possibly destroyed) objects.
  • Storing raw Thread.Sleep in game code: All updates happen on the main thread via the FNA game loop. Sleeping blocks the entire client.
  • Calling Profile.Save() every frame: JSON serialisation is expensive. Save only on meaningful user actions or at exit.
  • Bypassing SetScene(): Assigning to Client.Game.Scene directly skips the Dispose() / Load() lifecycle and will leak resources.
