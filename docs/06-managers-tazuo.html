TazUO Feature Managers

Session 06 — Automation, quality-of-life, and display systems unique to TazUO

Overview

TazUO adds a large layer of managers on top of ClassicUO's core systems. These managers handle features that were never present in the original client: automatic looting, property-based item highlighting, equipment durability tracking, message-triggered cooldown bars, overhead name filtering, and deeply configurable tooltip overrides.

All TazUO managers follow the same architecture as core managers (see Session 05) but tend to use EventSink subscriptions as their primary update mechanism rather than a polling Update() loop.

AutoLootManager TazUO

AutoLootManager automatically moves items from corpses (and the ground when scavenger mode is active) to the player's backpack or a configured "grab bag." It is a singleton that is created lazily via Instance and reset to null on scene unload.

Lifecycle

OnSceneLoad() ├── Load() — async Task.Factory.StartNew: reads AutoLoot.json into _autoLootItems ├── Subscribe EventSink.OPLOnReceive ├── Subscribe EventSink.OnItemCreated / OnItemUpdated ├── Subscribe EventSink.OnOpenContainer └── Subscribe EventSink.OnPositionChanged OnSceneUnload() ├── Unsubscribe all events ├── Save() — writes AutoLoot.json └── Instance = null — reset singleton

Event Triggers

EventHandlerWhat it does
OnItemCreated / OnItemUpdatedOnItemCreatedOrUpdatedIf item is in a corpse → check loot list. If ground item & scavenger enabled → check loot list.
OPLOnReceiveOnOPLReceivedWhen OPL data arrives for an item in a corpse, re-check the loot list (regex matching uses OPL property text).
OnOpenContainerOnOpenContainerWhen player opens a container, check if it is a corpse and queue eligible items.
OnPositionChangedOnPositionChangedRe-check nearby corpses each time the player moves (handles "walked away then back" case). Scavenger scan throttled to every 500 ms.
Why OnPositionChanged? Without this handler, if a player opens a corpse, walks a few tiles away, then walks back, no new events fire — the items would never be looted. The position-change trigger re-evaluates all nearby corpses on each step.

Loot Queue

Items to loot are placed into a static Queue<(uint item, AutoLootConfigEntry entry)>. The Update() method (called each frame from GameScene) dequeues one item per MoveMultiObjectDelay milliseconds, preventing item-spam rejection by the server.

// Dequeue one item per delay tick
if (_nextLootTime > Time.Ticks) return;

(uint item, AutoLootConfigEntry entry) = _lootItems.Dequeue();
// Resolve destination: entry.DestinationContainer → GrabBag → Backpack
ObjectActionQueue.Instance.Enqueue(new MoveRequest(moveItem.Serial, destinationSerial, ...).ToObjectActionQueueItem(), ActionPriority.LootItem);
_nextLootTime = Time.Ticks + ProfileManager.CurrentProfile.MoveMultiObjectDelay;

AutoLootConfigEntry — Match Logic

FieldTypeHow it's used
GraphicintMust match item graphic; -1 = any graphic
Hueushortushort.MaxValue = ignore hue; otherwise must match exactly
RegexSearchstringIf non-empty, matched against OPL name+data using RegexHelper.GetRegex() (cached, compiled)
DestinationContaineruintSerial of a specific container to loot into; 0 = use GrabBag or Backpack
UidstringAuto-generated Guid; used as stable key for removal

Human Corpse Protection

If ProfileManager.CurrentProfile.AutoLootHumanCorpses is false, human (player) corpses are skipped entirely. After looting, if HueCorpseAfterAutoloot is true, the corpse is hued to color 73 (grey) as a visual marker.

Scavenger Mode

When EnableScavenger is true, ground items (not in containers, not locked, distance ≤ 3 tiles) are also checked against the loot list every 500 ms on position change. This covers ground-spawned items that are never inside a corpse.

Import / Export

AutoLoot config is saved to {ProfilePath}/AutoLoot.json using AutoLootJsonContext (source-generated). The manager supports export/import by file and by clipboard JSON, and can read configs from other character profiles for cross-character copying.

HealthLinesManager TazUO

HealthLinesManager renders colored HP bars above every visible mobile. It is a pure draw system — no state is owned between frames. GameScene calls Draw(batcher) once per frame after the world render passes.

Display Modes

Profile SettingMeaning
ShowMobilesHP = falseNo bars drawn at all
MobileHPType = 0Show only text HP labels (no graphic bar)
MobileHPType = 1Show only graphic HP bar (no text)
MobileHPType = 2Show both text and graphic bar
MobileHPShowWhen = 0Always show
MobileHPShowWhen = 1Show only when HP < max
MobileHPShowWhen = 2Show only when HP ≠ max (same as 1)
HealthLineSizeMultiplierScale factor; bar width = 34 × multiplier pixels

Notoriety-Colored Bars

Bar fill color is derived from the mobile's NotorietyFlag via Notoriety.GetHue(). Blue = innocent, green = ally, gray = gray, orange = criminal, red = murderer. Passive targets (not last-target, not selected) are drawn at 50% alpha.

New Target System Integration

When UseNewTargetSystem is enabled and a mobile matches TargetManager.NewTargetSystemSerial, a special multi-gump overlay (graphic IDs 0x756A–0x756F) is drawn centered on the mobile, sized proportionally to the mobile's animation width. This is the "target ring" visual.

// Select correct sized target ring gumps based on animation width
if (width >= 80) { topGump = 0x756D; bottomGump = 0x756A; }
else if (width >= 40) { topGump = 0x756E; bottomGump = 0x756B; }
else              { topGump = 0x756F; bottomGump = 0x756C; }
ToList() on every draw call The inner loop uses _world.Mobiles.Values.ToList() to snapshot the collection before iterating. This prevents ConcurrentModificationException if a mobile is removed mid-frame by a network packet, but allocates a list every frame. This is the same defensive pattern used by DurabilityManager and other draw systems.

Viewport Clipping

Before drawing, screen position is checked against camera bounds. Bars outside the visible viewport are skipped without rendering — no off-screen draw calls.

GridHighlightData TazUO

GridHighlightData lives in Game/UI/Gumps/GridHighLight/GridHighLightData.cs. It provides property-based item highlighting: items displayed in grid containers are colored or marked when their OPL properties match a configured rule.

Rule Structure

Each GridHighlightSetupEntry (stored in Profile.GridHighlightSetup) defines one rule:

FieldPurpose
NameHuman-readable rule name, shown in tooltip when GridHighlightShowRuleName is true
ItemNamesList of item names this rule applies to; empty = all items
Hue / HighlightColorColor to render the grid cell background
PropertiesList of GridHighlightProperty entries; each has property name + min/max value
AcceptExtraPropertiesIf true, item can have properties beyond the listed ones and still match
MinimumMatchingProperty / MaximumMatchingPropertyHow many properties must match for the rule to fire
ExcludeNegativesProperty names that disqualify the item if present (e.g. "Lower Defense")

Background Matching

Matching is expensive (string ops + OPL data lookup), so it runs on a background task gated by a SemaphoreSlim(1, 1) to prevent concurrent access to the per-rule _normalizeCache dictionary. Items are queued via a static Queue<uint> and consumed by the background task.

Integration with Tooltip

ToolTipOverrideManager.BuildTooltip() calls GridHighlightData.GetBestMatch(itemPropertiesData) to find the highest-priority matching rule, then prefixes matched property lines with [o] to mark them in the tooltip text.

// In ToolTipOverrideManager.BuildTooltip()
GridHighlightData bestMatch = ProfileManager.CurrentProfile.GridHighlightProperties
    ? GridHighlightData.GetBestMatch(itemPropertiesData)
    : null;

if (isHighlighted)
    finalLine = $"[o] {finalLine}/cd"; // wrap matched property in highlight markup

DurabilityManager TazUO

DurabilityManager parses equipment durability values from OPL tooltip data and tracks them in a Dictionary<uint, DurabiltyProp>. The DurabilitysGump window reads this data to render a wear bar for each equipped item.

Data Flow

EventSink.OPLOnReceive fires └── OnOPLReceive(OPLEventArgs e) ├── Skip if not an Item serial ├── Skip if item is destroyed ├── If item is equipped on player in a tracked layer: │ └── ParseDurability(serial, e.Data) │ └── Regex: @"(?<=Durability )(\d*) / (\d*)" │ └── Returns DurabiltyProp(serial, current, max) │ └── _itemLayerSlots[item.Serial] = durability ├── Else: _itemLayerSlots.Remove(item.Serial) └── UIManager.ForEach<DurabilitysGump>(g => g.RequestUpdateContents())

Tracked Layers

The 21 equipment layers tracked are: Cloak, Shirt, Pants, Shoes, Legs, Arms, Torso, Tunic, Ring, Bracelet, Gloves, Skirt, Robe, Waist, Necklace, Beard, Earrings, Helmet, OneHanded, TwoHanded, Talisman.

The .ToList() Pattern

The Durabilities property exposes _itemLayerSlots.Values.ToList(). This is a deliberate defensive snapshot — any code reading durability data gets an immutable list that won't throw if the dictionary is modified by an incoming OPL packet on the network thread.

// Safe snapshot — callers can iterate without holding a lock
public List<DurabiltyProp> Durabilities => _itemLayerSlots.Values.ToList();

DurabiltyProp

PropertyTypeMeaning
SerialintItem serial this durability record belongs to
DurabiltyintCurrent durability value
MaxDurabiltyintMaximum durability value
PercentagefloatComputed: Durabilty / MaxDurabilty, 0 if max is 0

HasDurabilityData is a static bool set to true whenever the dictionary has any entries — gumps can check this to avoid rendering an empty durability panel.

CoolDownBarManager TazUO

CoolDownBarManager creates on-screen cooldown bar gumps when specific text messages are received. Up to 15 simultaneous cooldown bars are supported (MAX_COOLDOWN_BARS = 15), tracked in a static array CoolDownBar[].

Trigger Logic

Each trigger condition is stored in the Profile as parallel lists: Condition_Trigger[], Condition_Duration[], Condition_Label[], Condition_Hue[], Condition_Type[]. On every MessageReceived event, all conditions are checked:

private void MessageManager_MessageReceived(object sender, MessageEventArgs e)
{
    int count = ProfileManager.CurrentProfile.CoolDownConditionCount;
    for (int i = 0; i < count; i++)
    {
        // Condition_Type: 0=any, 1=self only, 2=others only
        switch (ProfileManager.CurrentProfile.Condition_Type[i]) { ... }

        if (e.Text.Contains(ProfileManager.CurrentProfile.Condition_Trigger[i]))
            AddCoolDownBar(World, TimeSpan.FromSeconds(duration), label, hue, replace);
    }
}

Bar Placement

BehaviorDetail
Slots 0–14First available empty/disposed slot is used; bars stack vertically by index × (height + 5)
Replace modeIf Condition_ReplaceIfExists[i] is true and a bar with the same label exists, it is disposed and re-created in the same slot (resets timer)
PositionDefault X/Y defined by CoolDownBar.DEFAULT_X / DEFAULT_Y constants
Use case example A server sends "You begin casting Explosion" as a system message. A cooldown condition triggers on that text, showing a 5-second bar labeled "Explosion." When the bar expires, the CoolDownBar gump disposes itself automatically.

WorldDataManager TazUO

WorldDataManager supports server-driven world generation: servers can send custom terrain and static data that overrides the client's UO map files. This is TazUO's procedural generation infrastructure.

Components

ComponentRole
WorldDataStoreIn-memory grid of WorldChunkData objects, keyed by block X×mapHeight+Y
WorldDataPersistenceHandles None / FileBacked persistence modes; FileBacked mode saves chunks to procgen/{worldId}/
WorldChunkData64 land graphics + 64 Z values + list of ServerStaticEntry items for one 8×8 block

Persistence Modes

ModeBehavior
NoneServer data is held in RAM only; lost on logout. Used for temporary dynamic events.
FileBackedEach received block is immediately flushed to disk at procgen/{worldId}/. On next login, chunks are loaded from disk instead of the UO map files.

Server API

// Called by packet handlers when server sends terrain data
manager.ReceiveTerrainBlock(blockX, blockY, graphics, zValues);

// Called when server sends statics for a block
manager.ReceiveStaticsBlock(blockX, blockY, statics);

// Called to drop server data and fall back to UO files
manager.ReleaseBlock(blockX, blockY);

Chunk Invalidation

After storing new data, InvalidateChunk() looks up the corresponding Chunk in the live map and calls Destroy() on it. The map's normal chunk loading then reloads the chunk, this time finding WorldDataManager's data and using it instead of the mul file.

View Tracking

UpdateViewBounds() is called by GameScene every frame after computing the viewport. When the visible block range changes, it fires ViewBoundsChanged — allowing server-side systems to know which blocks are visible and stream only the relevant terrain.

NameOverHeadManager TazUO

NameOverHeadManager controls which entities show names floating above them. Instead of a simple on/off toggle, TazUO implements a full option set system where multiple named configurations can be defined and activated by hotkey.

IsShowing State

// Three ways names can be visible
public static bool IsShowing =>
    IsPermaToggled          // toggled on permanently via menu/macro
    || IsTemporarilyShowing  // hotkey held down
    || Keyboard.Ctrl && Keyboard.Shift;  // built-in Ctrl+Shift shortcut

NameOverheadOptions Flags

The NameOverheadOptions enum is a [Flags] bitmask. Each NameOverheadOption stores an int bitmask of which entity types/notorieties to show:

CategoryFlags
ItemsContainers, Gold, Stackable, LockedDown, Moveable, Immoveable, Other
CorpsesMonsterCorpses, HumanoidCorpses, OwnCorpses
Mobile typeHumanoid, Monster, OwnFollowers, Self, ExcludeSelf
Mobile notorietyInnocent, Ally, Gray, Criminal, Enemy, Murderer, Invulnerable

Options — Named Presets

Options are saved to {ProfilePath}/nameoverhead.xml. Default presets created on first run:

NameFlags
Allint.MaxValue — everything
Mobiles onlyAllMobiles
Items onlyAllItems
Mobiles & Corpses onlyMobilesAndCorpses

IsAllowed() — The Filter

Called by the render system for every entity being drawn. Returns true only if the active NameOverheadOptions bitmask permits the entity's type and notoriety:

public bool IsAllowed(Entity entity)
{
    if (SerialHelper.IsItem(entity)) return HandleItemOverhead(entity);
    if (SerialHelper.IsMobile(entity)) return HandleMobileOverhead(entity);
    return false;
}

Hotkey Activation

RegisterKeyDown(key, mod) looks up the matching NameOverheadOption by key+modifiers, sets ActiveOverheadOptions, and sets IsTemporarilyShowing = true. On RegisterKeyUp(), IsTemporarilyShowing = false. This "press and hold" behavior is how the Ctrl+Shift built-in shortcut was extended to user-defined hotkeys.

ToolTipOverrideManager TazUO

ToolTipOverrideData / ToolTipOverrideManager lets players define rules that reformat item tooltip text. For example, the raw line "Weapon Damage 15 to 18" can be rewritten as a colored, compact "DMG /c[orange]15/cd - /c[red]18".

Override Rule Structure

FieldMeaning
SearchTextText to match in the tooltip property line. Prefix $ for regex matching.
FormattedTextFormat string; {0}=property name, {1}=first value, {2}=second value, {3}=original text, {4}=first diff, {5}=second diff
Min1 / Max1Numeric range filter for the first value in the property line
Min2 / Max2Numeric range filter for the second value
ItemLayerTooltipLayers enum: Any, specific Layer enum value, or group (Body_Group, Jewelry_Group, Weapon_Group)

BuildTooltip() Pipeline

ProcessTooltipText(world, serial) └── BuildTooltip(ItemPropertiesData) ├── 1. Handle item name line │ ├── Try override: SearchText matches item name → use FormattedText │ └── Fallback: apply TooltipHeaderFormat from Profile ├── 2. Find best GridHighlightData rule (if GridHighlightProperties=true) └── 3. For each property in itemPropertiesData.singlePropertyData: ├── Check if property is highlighted by GridHighlight rule ├── Try override: SearchText matches property text AND values in Min/Max range │ ├── Match: format with FormattedText │ └── No match: use original property text └── If highlighted: prefix with "[o]", suffix with "/cd"

Layer Filtering

Overrides can be scoped to specific equipment layers to prevent one rule from affecting multiple item types. Layer groups simplify common cases:

  • Body_Group — Shoes, Pants, Shirt, Helmet, Necklace, Arms, Gloves, Waist, Torso, Tunic, Legs, Skirt, Cloak, Robe
  • Jewelry_Group — Talisman, Bracelet, Ring, Earrings
  • Weapon_Group — OneHanded, TwoHanded

Storage in Profile

Override settings are stored as parallel lists in the Profile JSON (not a list of objects). ToolTipOverride_SearchText[i], ToolTipOverride_NewFormat[i], ToolTipOverride_MinVal1[i], etc., all share the same index. ToolTipOverrideData.Get(index) assembles these into a transient object on demand.

Cache invalidation required ToolTipOverrideData maintains a _cachedOverrides array keyed by count. After any Save() or Delete(), InvalidateCache() must be called so the next GetAllToolTipOverrides() rebuilds the array. Forgetting this results in stale override data being shown.

SpellBarManager TazUO

SpellBarManager is a static class that maintains a configurable grid of spell shortcuts ("spell bar") supporting keyboard hotkeys and game controller buttons.

Data Model

TypeRole
SpellBarRow10 spell slots; stores spell IDs as int[] for serialization, exposes SpellDefinition[] lazily
SpellBarRowsStatic List<SpellBarRow>; multiple rows allow different loadouts
SpellBarSettingsEnabled flag, hotkey mappings (SDL keycodes + modifiers), and controller button combos for each slot
CurrentRowActive row index; can be switched via UI to change which loadout is active

Key Matching

On KeyPress(key, mod), left/right modifier variants are normalized to generic (e.g. SDL_KMOD_LCTRL + SDL_KMOD_RCTRL → SDL_KMOD_CTRL). This ensures bindings work regardless of which physical key is pressed.

Persistence

Saved to {ProfilePath}/SpellBar.json and {ProfilePath}/SpellBarSettings.json using source-generated JSON contexts. Presets (per-row saved configurations) are stored in Data/SpellBarPresets/ as individual JSON files.

TazUO-Specific Patterns

TazUO managers share several implementation patterns that distinguish them from ClassicUO's original managers:

1. EventSink-First Architecture

TazUO managers subscribe to EventSink events rather than polling every frame. AutoLootManager subscribes to 4 events; DurabilityManager subscribes to 1. This is more efficient and ensures the feature only runs when something actually happens.

PatternClassicUO styleTazUO style
Update triggerUpdate() called every frameEventSink subscription fires on data change
Collection iterationDirect iteration, occasional unsafe.ToList() snapshot before iterating across any collection that network packets might modify
Save formatXMLJSON with source-generated JsonSerializerContext
Singleton patternStatic readonly fieldsstatic Instance { get { field ??= new(); return field; } } with reset on unload
Load timingSynchronous in constructorTask.Factory.StartNew() for async load (AutoLoot); _loaded guard prevents processing before load completes

2. Source-Generated JSON Serialization

Every manager that persists data uses a [JsonSerializable]-annotated JsonSerializerContext subclass. This is required for .NET NativeAOT/trimming compatibility and is enforced project-wide:

// Correct pattern
[JsonSerializable(typeof(AutoLootManager.AutoLootConfigEntry))]
[JsonSerializable(typeof(List<AutoLootManager.AutoLootConfigEntry>))]
[JsonSourceGenerationOptions(WriteIndented = true)]
public partial class AutoLootJsonContext : JsonSerializerContext { }

// Usage
JsonHelper.Load(path, AutoLootJsonContext.Default.ListAutoLootConfigEntry, out _autoLootItems);
Never use reflection-based JSON JsonSerializer.Serialize(obj) without a type context will compile but fail at runtime in trimmed/NativeAOT builds. Always use the context: JsonSerializer.Serialize(obj, MyContext.Default.MyType).

3. The ObjectActionQueue Pattern

TazUO managers that send game actions (move item, use item, etc.) go through ObjectActionQueue.Instance.Enqueue() rather than calling GameActions.* directly. This queue enforces action spacing and priority:

PriorityUsed by
ActionPriority.LootItemAutoLootManager
ActionPriority.NormalMost user interactions
ActionPriority.HighCritical actions (combat target)

4. Profile as the Configuration Store

Where ClassicUO managers often had their own config files, TazUO managers increasingly store their settings directly in the per-character Profile. This simplifies the persistence story (one JSON file per character) but makes the Profile extremely large (~500 properties).

The exception is data that is too large or independently useful: AutoLoot.json, nameoverhead.xml, and SpellBar.json remain separate files.

5. Async Load with Guard Flag

AutoLootManager illustrates a pattern used for heavy loads: start an async task, set _loaded = true when done, and guard all processing methods with if (!_loaded) return;. This prevents the manager from acting on events before its config is ready.

Going Forward

Adding a New TazUO Feature Manager

  1. Create the class in Game/Managers/. Use constructor injection for World.
  2. Register in World as a property. World is re-instantiated on each login, so all managers are fresh.
  3. Subscribe events in OnSceneLoad(), unsubscribe in OnSceneUnload(). Never subscribe in the constructor — the scene may not exist yet.
  4. Use source-generated JSON for any persistence. Add a [JsonSerializable] context class next to the data class.
  5. Use ObjectActionQueue for any game actions, never call GameActions.* from a tight loop or event handler directly.
  6. Snapshot collections before iterating — any collection that a network thread might modify must be snapshotted with .ToList() or a lock before iteration.
