Rendering Pipeline

How TazUO converts world-state into pixels: the four render lists, isometric depth sorting, TNext tile traversal, the UltimaBatcher2D sprite batcher, camera math, and how UO asset files are loaded and cached.

Pipeline Overview

TazUO renders a 2D isometric world using FNA's GraphicsDevice. There is no Z-buffer — draw order is determined entirely by the tile traversal order, and must never be changed by sorting by texture. The pipeline runs entirely on the main thread.

GameController.Draw(GameTime) │ └─ GameScene.Draw(UltimaBatcher2D batcher) │ ├─ PHASE 1 — Sort & populate render lists │ GameScene.SortAndAddToRenderList() │ └─ for each visible tile column (NW→SE diagonal sweep): │ AddTileToRenderList(x, y, maxZ) │ └─ walk TNext linked list at chunk.Tiles[x%8, y%8] │ → classify each object into one of 4 lists │ ├─ PHASE 2 — Render to _worldRenderTarget │ batcher.Begin(...) │ DrawRenderList(_renderListStatics) ← terrain + static tiles │ DrawRenderList(_renderListAnimations) ← mobiles + items on ground │ DrawRenderList(_renderListEffects) ← spell effects, particles │ DrawRenderList(_renderListTransparentObjects) ← semi-transparent objects │ batcher.End() │ ├─ PHASE 3 — Apply lighting │ Render lights to _lightRenderTarget │ Composite with darkness blend mode │ ├─ PHASE 4 — Post-process & composite to backbuffer │ Apply XBR / post-processing effect (if enabled) │ Draw _worldRenderTarget scaled to screen │ └─ UIManager.Draw(batcher) ← gumps drawn on top of world
Immutable draw order
The isometric renderer relies on painter's algorithm — objects are drawn back-to-front, with objects farther from the camera drawn first. This order is determined by the NW→SE tile sweep and the TNext list order within each tile. You must never sort the render lists by texture, sprite sheet, or any other criterion. Doing so will break visual layering (objects will appear to float or be buried underground).

The Four Render Lists

Objects collected during tile traversal are classified into one of four List<GameObject> instances maintained as fields of GameScene:

ListContentsWhy separate?
_renderListStatics Land tiles, Static world decorations, non-animated Items, Multi pieces Opaque geometry drawn first; texture atlased for batch efficiency
_renderListAnimations Mobiles (players, NPCs, monsters), animated Items Need animation frame lookup from Animations loader each draw; separate pass allows animation state update
_renderListEffects Spell effects, projectiles, particle systems, lightning Drawn above animations; often use additive or alpha blending separate from world objects
_renderListTransparentObjects Foliage being faded, objects with AlphaHue < 255, roofs being hidden Require alpha blending — must be drawn after all opaque objects in correct back-to-front order

The lists are cleared at the start of each frame and repopulated by AddTileToRenderList(). Objects with AllowedToDraw == false or IsDestroyed == true are skipped during classification.

Isometric Depth Sorting

Ultima Online uses a classic isometric projection where "north-west" is the top of the screen and "south-east" is the bottom. To achieve correct painter's algorithm ordering, tiles must be visited in the order that their projected pixels appear from the back of the scene to the front.

World tile axes: X → east, Y → south Projected to screen: screenX = (x - y) * TILE_SIZE_HALF (horizontal position) screenY = (x + y) * TILE_SIZE_HALF (vertical position, before Z offset) screenY -= z * 4 (Z height adjustment) Tile draw order — diagonal column sweep (NW→SE): For diagonal d = 0 to maxD: For each tile on diagonal d where x - y == d (constant): visit tile (x, y) — tiles to NW of player are visited before tiles to SE Objects within the same tile cell are ordered by: 1. Z value (lower Z → drawn first, higher Z on top) 2. Object type priority (Land < Static < Item < Mobile)

Why You Cannot Sort by Texture

A common optimisation in 2D games is to sort draw calls by texture to reduce GPU state changes (texture switches). This is impossible in TazUO because:

  • The art atlas is not fully deterministic — sprites from different tile types land on different textures.
  • Reordering any two sprites that share a screen-space overlap would show through incorrectly.
  • Even tiles that appear non-overlapping at low zoom levels overlap at higher zoom or in tall structures.

The only performance lever for batching is reducing the number of texture switches within each render list — which is why the four lists exist (statics are mostly atlas-batched; animations always require individual texture lookups).

TNext Linked-List Traversal

Every tile cell in a Chunk holds the head of a singly-linked list of GameObject instances via the TNext property (from LinkedObject). The render code walks this list for every tile in the view frustum. 

// Classic TNext traversal pattern (from GameSceneDrawingSorting.cs)
for (
    GameObject obj = chunk.GetHeadObject(x, y), _tnext = null;
    obj != null;
    obj = _tnext
)
{
    _tnext = obj.TNext;   // capture BEFORE any operation that might modify the list

    if (obj.IsDestroyed)  // guard — object may have been destroyed mid-frame
        continue;

    // classify obj into one of the 4 render lists
    AddToRenderList(obj);
}
Always capture TNext first
The pattern _tnext = obj.TNext before any processing is critical. If processing modifies the linked list (e.g. removing a destroyed object), reading TNext after modification would give the wrong next node or null-deref. Capture it immediately after entering the loop body.

UltimaBatcher2D — The Sprite Batcher

src/ClassicUO.Renderer/Batcher2D.cs — UltimaBatcher2D is the only path to the GPU. Every sprite drawn to the world goes through its Draw*() overloads. It buffers up to MAX_SPRITES = 0x800 (2048) sprites per batch before flushing to the GPU. 

// Core internal state
private const int MAX_SPRITES  = 0x800;   // 2048 sprites max per batch
private const int MAX_VERTICES = MAX_SPRITES * 4;
private const int MAX_INDICES  = MAX_SPRITES * 6;

DynamicVertexBuffer _vertexBuffer;     // GPU vertex buffer
IndexBuffer         _indexBuffer;      // pre-filled with quad indices
Texture2D[]         _textureInfo;      // one per sprite in batch
PositionNormalTextureColor4[] _vertexInfo; // 4 verts per sprite
private int         _numSprites;       // current fill count

Flush Triggers

The batcher flushes (submits to GPU) when:

  • The sprite count reaches MAX_SPRITES (2048)
  • The texture changes (UO sprites are not all on the same atlas)
  • End() is called at the end of a draw pass
  • Blend state, sampler, or effect changes (pass state change)

Custom UO Effect

UltimaBatcher2D uses BasicUOEffect — a custom shader that takes a hue vector (hueIndex, shaderMode, alpha) as vertex data. The GPU samples the hue texture (loaded into GraphicsDevice.Textures[1]) to colorize sprites without CPU-side pixel manipulation.

ShaderModeEffect
SHADER_NONENo hue applied — raw sprite colours
SHADER_HUEDApply hue from hue texture (most world objects)
SHADER_PARTIAL_HUEDHue only non-black pixels (equipment on mobiles)
SHADER_LANDLand tile rendering with texture blending
SHADER_LIGHTSLight rendering pass
SHADER_TEXT_HUEText rendering with colour override

Camera & Viewport

src/ClassicUO.Renderer/Camera.cs manages the view matrix, zoom level, and optional "peek" mode where the camera pans toward the mouse cursor.

PropertyTypePurpose
BoundsRectanglePixel rectangle of the game viewport within the window
ZoomfloatScale factor (clamped to ZoomMin–ZoomMax). 1.0 = native UO pixel size
ZoomStepfloatHow much each zoom in/out changes the zoom level (default 0.1)
OffsetVector2Lerped camera offset; smoothly follows the player or peek direction
PeekingToMouseboolWhen true, camera offset lerps toward the cursor (up to MAX_PEEK_DISTANCE = 250px)
ViewTransformMatrixMatrixCombined translation + scale matrix; passed to the batcher's Begin() call

The camera's Update(force, timeDelta, mousePos) method is called each frame. When peeking, the offset lerps toward the mouse direction at a rate proportional to distance, providing a smooth "look ahead" effect.

Coordinate Spaces

World tile (x, y, z) ↓ isometric projection Screen pixel (screenX, screenY) in world render target space ↓ Camera.ViewTransformMatrix (zoom + offset) Viewport pixel (viewX, viewY) in the game window ↓ RenderScale (GameController.SetScale) Final backbuffer pixel for display

The GameScene maintains _minPixel and _maxPixel vectors that define the frustum of visible tiles at the current zoom and offset. Only tiles whose projected positions fall within this range are visited by the tile sweep.

Draw Passes in Detail

Pass 1 — World Render Target

The world is rendered into _worldRenderTarget (a RenderTarget2D the size of the game viewport). This isolates the world from UI and allows post-processing. The batcher uses BlendState.AlphaBlend and SamplerState.PointClamp for crisp pixel-art rendering.

Pass 2 — Lighting

Each light source (torches, spells, player light level) is collected into the _lights[] array during world object traversal. The light pass renders each light into _lightRenderTarget using additive blending, then composites the light texture over the world using a custom darkness blend: 

// _darknessBlend — multiplies world by light mask
ColorSourceBlend      = Blend.Zero
ColorDestinationBlend = Blend.SourceColor
ColorBlendFunction    = BlendFunction.Add

// _altLightsBlend — add bright light on top
ColorSourceBlend      = Blend.DestinationColor
ColorDestinationBlend = Blend.One

Pass 3 — Post-Processing

If XBR scaling or a post-processing effect is enabled (_filterMode), the world render target is passed through a GLSL/HLSL effect before being composited. Point sampling (no filtering) is used by default for authentic pixel-art look.

Pass 4 — UI Layer

After the world passes, UIManager.Draw() renders all gumps directly to the backbuffer (not through the world render target). ImGui is rendered last via ImGuiNET.ImGui.Render() before GraphicsDevice.Present().

Asset Loaders

Asset loading is split between the ClassicUO.Assets project (file parsing) and the ClassicUO.Renderer project (GPU upload). The pattern is consistent: the Assets layer reads the UO binary file and produces raw pixel data; the Renderer layer uploads it to a Texture2D and caches the result.

Texture Atlas

src/ClassicUO.Renderer/TextureAtlas.cs packs multiple small sprites into a single large Texture2D to reduce texture switches. Art tiles are packed into atlases. When a tile is requested and not yet loaded, it's decoded from the UO file and uploaded to the atlas.

Animation frames are not atlas-packed because they vary greatly in size and are accessed per-frame with unpredictable access patterns. Each animation frame may be on its own texture, which is why animation rendering causes more texture switches than static tile rendering.

Performance Notes

HotspotMitigation in TazUO
Texture switches TextureAtlas packs art tiles together. Four render lists keep static/animation/effect batches separate. Flush triggers are minimised within each list.
TNext traversal per frame Only tiles in [_minTile, _maxTile] range are visited. UpdateMaxDrawZ() skips recalculation if player position hasn't changed.
Foliage transparency FoliageIndex batches foliage into groups by position. A cache (_surfaceOverheadCache) prevents re-checking the same position every frame.
Shader texel size Batcher2D caches the last set texel size and skips redundant Pass.Apply() calls when the texture size hasn't changed.
Roof culling UpdateMaxDrawZ() computes _maxZ — objects above this Z are not added to render lists when the player is indoors (no roofs drawn over player).

Going Forward

Safe Patterns

  • Adding a new visual effect: Create a GameEffect subclass, add it to the world via World._effectManager.Add(effect). It will automatically be classified into _renderListEffects.
  • Drawing custom HUD elements: Override Gump.Draw() and call batcher.DrawTexture() / batcher.DrawRectangle() etc. Gumps draw after the world and before ImGui.
  • Reading tile data for rendering logic: Use obj.ItemData (ref to StaticTiles) for statics, or land.TileData for terrain — these are zero-copy references into the preloaded file data.

Anti-Patterns to Avoid

  • Never sort render lists by texture: This breaks visual layering. Accept the texture switch cost — it's intentional.
  • Never draw directly to GraphicsDevice: Always go through UltimaBatcher2D. Direct GPU calls outside the batcher's Begin/End will corrupt the vertex buffer state.
  • Never hold Texture2D references long-term: The atlas may evict and re-upload textures. Always look up the texture from the loader at draw time.
  • Don't add per-object rendering logic to the sort phase: AddTileToRenderList runs for every tile every frame. Expensive operations (file I/O, allocations) in this method will tank frame rate. Keep classification cheap.
