Game Objects & World State

The object hierarchy from LinkedObject to PlayerMobile, how serial numbers identify every entity, how the World container owns everything, and how Map/Chunk spatially organises the tile grid.

Class Hierarchy

LinkedObject ← intrusive doubly-linked list node └─ BaseGameObject ← adds World reference + RealScreenPosition └─ GameObject ← Graphic, Hue, X/Y/Z, IsDestroyed, Distance ├─ Entity ← Serial, Hits, Name, Direction, Flags (IEquatable) │ ├─ Mobile ← Steps queue, animation, combat flags │ │ └─ PlayerMobile ← Skills[], Walker, Pathfinder, stats │ └─ Item ← Container, Amount, Layer, IsCorpse ├─ Land ← terrain tile: AverageZ, normals, stretching ├─ Static ← world decoration tile: ItemData, Index └─ Multi ← large structures (houses, ships) Also derives from GameObject (effects): GameEffect → MovingEffect, FixedEffect, LightningEffect, DragEffect

Only Entity subclasses have serials. Land, Static, and Multi are world-geometry objects with no server-assigned identity — they are created from the UO map files and never sent individual update packets.

Serial Numbers

Every server-tracked entity (Mobile and Item) has a 32-bit serial number. The serial encodes whether an entity is a mobile or an item: 

// SerialHelper.cs — the definitive serial rules
public static bool IsValid (uint serial) => serial > 0          && serial < 0x80000000;
public static bool IsMobile(uint serial) => serial > 0          && serial < 0x40000000;
public static bool IsItem  (uint serial) => serial >= 0x40000000 && serial < 0x80000000;
RangeEntity typeHelper
0x00000001 – 0x3FFFFFFFMobiles (NPCs, players, monsters)SerialHelper.IsMobile(s)
0x40000000 – 0x7FFFFFFFItems (equipment, containers, corpses)SerialHelper.IsItem(s)
0x00000000Null / invalidSerialHelper.IsValid(s) → false
≥ 0x80000000Server-internal / not a valid client serialSerialHelper.IsValid(s) → false

The local player's serial is stored in World.Player.Serial. You can compare any entity to the player with entity.Serial == world.Player — there is an implicit conversion from PlayerMobile to uint.

Lookup pattern
Given an arbitrary serial from a packet, use World.Get(serial) which searches both World.Mobiles and World.Items. For typed lookup use World.Mobiles.TryGetValue(serial, out Mobile m) or World.Items.TryGetValue(serial, out Item item).

Animation System on Mobile

Animation state is managed across several fields used by the rendering system:

  • _animationGroup (byte) — which animation group (walk, run, attack, die, etc.)
  • AnimIndex — current frame index within the group
  • _animationInterval — ticks between frame advances
  • _animationRepeat / _animationRepeatModeCount — loop control
  • LastAnimationChangeTime — last tick when frame advanced

The rendering code calls Mobile.ProcessAnimation() each frame to advance the frame counter. The render pass then calls Animations.GetAnimationFrames() with the current group + frame to get the correct sprite.

Land, Static, and Multi

These three classes represent the non-interactive world geometry. They have no serial numbers and are never individually updated by the server after initial map load.

ClassSourceKey fieldsNotes
Land UO map file (terrain) AverageZ, MinZ, IsStretched, NormalTop/Right/Left/Bottom, YOffsets Terrain tiles. Stretching applies when neighbouring tiles have different heights — normals used for lighting.
Static UO statics file (decoration) ItemData (ref), Index, IsVegetation World decorations. ItemData is a zero-copy ref into the TileData static array.
Multi Multi file (houses, ships) MultiInfo, component items Large multi-tile structures. Represented as a single logical object with a bounding region.

All three are created by Chunk.Load() when a map block is loaded. They live in the per-tile linked list (the TNext chain) inside Chunk.Tiles[x, y].

PlayerMobile — The Local Player

Game/GameObjects/PlayerMobile.cs extends Mobile with everything specific to the character the user is playing. There is exactly one instance, referenced by World.Player. 

public class PlayerMobile : Mobile
{
    public Skill[] Skills { get; }               // one per skill, loaded from skill file
    public WalkerManager Walker { get; }         // client-side movement prediction
    public Pathfinder Pathfinder { get; }        // A* auto-walk
    public IReadOnlyDictionary<BuffIconType, BuffIcon> BuffIcons { get; }
    public readonly Ability[] Abilities;          // [0]=Primary, [1]=Secondary

    // Stat fields set from 0x11 / 0x3A packets
    public short ColdResistance, FireResistance, EnergyResistance;
    public short DamageMin, DamageMax, DamageIncrease;
    public short DefenseChanceIncrease, HitChanceIncrease;
    // ... (many more stat fields)

    // Corpse tracking (for auto-open logic)
    public readonly HashSet<uint> AutoOpenedCorpses;
    public readonly HashSet<uint> ManualOpenedCorpses;
}

PlayerMobile also overrides InWarMode (settable from the player, only computed on other mobiles) and adds IsVisible (always true for the player, can be toggled for stealth effects).

World — The Global Container

Game/World.cs is the single source of truth for all in-game state. It owns the entity dictionaries, the map, and all the managers. 

public sealed class World
{
    // Entity registries (main thread access only)
    public Dictionary<uint, Item>   Items   { get; }
    public Dictionary<uint, Mobile> Mobiles { get; }

    // The local player; null between scenes
    public PlayerMobile Player { get; private set; }

    // Current map; null before entering world
    public Map.Map Map { get; private set; }

    // Convenience
    public bool   InGame          => Player != null && Map != null;
    public byte   ClientViewRange { get; set; }  // from profile ViewRange
    public Season Season          { get; private set; }
    public int    MapIndex         { get; set; }   // triggers map reload
}

Corpse Fast Path

In addition to the main Items dictionary, World maintains a HashSet<Item> _corpses (with lock _corpsesLock) for O(1) corpse enumeration. Use World.GetCorpseSnapshot() to get a thread-safe array copy before iterating.

MapIndex Setter — Map Reload

Setting World.MapIndex triggers a full map reload: the old Map is destroyed, a new Map is created, a new WorldDataManager is created for the map dimensions, and the player's position is re-anchored. This is called from network packet handlers when the server sends a map change.

Map & Chunk — Spatial Organisation

The world is divided into 8×8-tile chunks. The Map object manages a dictionary of loaded chunks and requests chunk loads as the player moves.

World tile space (e.g. 7168 × 4096 for Felucca / Trammel) │ └─ divided into chunks: tile(x,y) → chunk(x/8, y/8) │ Chunk[8][8] grid of GameObject (first in TNext linked list) │ Each Chunk.Tiles[cx, cy] is the HEAD of a linked list of all GameObjects at tile (chunkX*8+cx, chunkY*8+cy): Land → Static → Static → ... → Item → Item → ... (ordered by Z for rendering)

Chunk Load Priority

Chunk.Load() checks sources in priority order:

  1. WorldDataManager in-memory store — TazUO world edits override map files
  2. WorldDataManager file-backed store — persisted edits from previous sessions
  3. UO map files — the original Ultima Online data

Object Lifecycle

Network packet received (on main thread after queue dispatch) │ ├─ CREATE path (e.g. 0x78 Mobile Update / 0x1A Item Update) │ PacketHandlers.MobileUpdate() / ItemUpdate() │ ├─ World.Mobiles.TryGetValue() — existing? update fields │ └─ else: new Mobile(world, serial) → World.Mobiles.Add() │ → World.Map.GetTile(x, y).AddGameObject(mobile) │ (inserts into the TNext linked list) │ ├─ UPDATE path (e.g. 0x77 Mobile Moving) │ entity.X = newX; entity.Y = newY; entity.Z = newZ; │ entity.IsPositionChanged = true; │ (chunk sorts object into correct tile slot next frame) │ └─ DESTROY path (e.g. 0x1D Delete Object) World._toRemove.Add(serial) └─ World.Update() calls RemoveMobile/RemoveItem: entity.Destroy() ← sets IsDestroyed = true entity.RemoveFromTile() ← unlinks from TNext chain World.Mobiles/Items.Remove(serial)
IsDestroyed is a hard stop
Any code that holds a reference to a GameObject must check obj.IsDestroyed before accessing any field. Destroyed objects are not immediately GC'd — they can remain referenced in lists, manager caches, or gumps until those references are cleared. Accessing a destroyed object's X/Y/Z or Graphic after Destroy() may return stale or zeroed data.

Deletions are deferred to World.Update() via the _toRemove list. This prevents modification of World.Mobiles/World.Items during iteration — a common source of InvalidOperationException bugs.

Going Forward

PatternSafe approach
Accessing any game object Always check obj != null && !obj.IsDestroyed first, even if you just retrieved it from a dictionary.
Iterating Mobiles or Items Use .ToList() or .ToArray() on the values collection before iterating if you might add/remove during the loop. Or use _toRemove deferred deletion.
Finding an entity by serial World.Get(serial) returns Entity (checks both collections). For typed access use World.Mobiles.TryGetValue or World.Items.TryGetValue.
Adding TazUO fields to Item Add them directly to Item.cs. Reset them in the pool-reset code (the commented-out block) if pooling is re-enabled.
Storing item references in managers Store the uint serial, not the Item reference. Look up the object by serial when needed. Stale object references are a major crash source.
Thread safety
World.Mobiles and World.Items are plain Dictionary instances — not thread-safe. Only the main game thread should read or write them. The network thread queues updates as packets, which are dispatched on the main thread. The corpse collection uses a dedicated lock for the one cross-thread use case.
